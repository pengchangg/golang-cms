# 企业内部 CMS 多 Agent 开发计划

## 1. 文档状态

- 状态：已规划，待执行
- 关联设计：[企业内部 CMS 设计文档](./cms-design.md)
- 推荐执行规模：4 个实现 Agent + 1 个集成 Agent
- 交付策略：三个可独立部署的功能阶段
- 协作策略：固定领域、固定目录所有权、分层冻结契约、统一集成门禁

本文档用于指导多个 Agent 并行实施 CMS。产品范围、技术选型和验收目标以 `docs/cms-design.md` 为准；本文只定义开发步骤、依赖、职责、合并顺序和验证方法。

## 2. 执行原则

### 2.1 基本原则

- 先冻结共享契约，再并行实现。
- 每个 Agent 固定负责一个领域，不使用自由领取任务模式。
- 业务 Agent 不直接修改应用入口、OpenAPI 根文件和迁移执行器。
- 跨模块访问必须通过公开服务接口，禁止跨模块直接读写数据表。
- 每个任务包必须可独立编译和测试。
- 未完成的功能不注册运行时路由，不使用返回成功的占位实现。
- 阶段一即采用不可变内容版本，不先做原地更新再迁移。
- 每个阶段完成后都必须形成可独立部署和使用的系统。
- 已合入或可能执行过的数据库迁移不得修改，只能增加向前兼容迁移。
- OpenAPI 是前后端共享契约，不是项目结束时补写的文档。

### 2.2 推荐分工

| Agent | 负责领域 | 核心产出 |
| --- | --- | --- |
| Agent 1 | 平台、安全、身份、权限、审计 | 配置、数据库基础、OIDC、会话、RBAC、审计、任务运行基础 |
| Agent 2 | 动态模型、内容、工作流、CSV | Schema、Revision、投影、关联、审核流、导入导出 |
| Agent 3 | 客户端 API、API Key、OSS | 内容 REST API、API Key、素材库、OSS Adapter |
| Agent 4 | React 管理端 | 所有管理页面、动态表单、前端测试 |
| 集成 Agent | 契约、迁移、装配、发布门禁 | 应用入口、路由、OpenAPI 聚合、Docker、集成测试、E2E |

## 4. Git 和协作方式

### 4.1 基线要求

当前项目需要先建立基线提交，再开始并行开发。基线至少包含：

- `docs/cms-design.md`。
- `docs/development-plan.md`。
- `.gitignore`。
- `README.md`。
- Go、React、迁移和 OpenAPI 骨架。
- 基础测试命令。
- 目录所有权说明。

没有基线提交时，不应创建并行分支或 Git worktree。

### 4.2 分支建议

```text
main
integration/v1
agent/platform-security
agent/content-schema
agent/client-assets
agent/admin-web
```

每个任务包使用短期分支：

```text
agent/platform-security/p1-auth
agent/content-schema/p1-models
agent/client-assets/p2-api-key
agent/admin-web/p1-model-editor
```

### 4.3 合并规则

- 功能分支先合入 `integration/v1`。
- 每个阶段验收完成后，再将 `integration/v1` 合入 `main`。
- Agent 不直接合并其他 Agent 的分支。
- 一个任务包只包含本任务需要的改动。
- 共享契约变更必须先由集成 Agent确认，不能在功能 PR 中静默修改。
- 集成故障退回对应模块 Agent 修复，集成 Agent 不重写业务逻辑。
- 不得通过跳过测试、降低门禁或删除约束解决合并阻塞。

### 5.2 Agent 2：动态模型、内容、工作流与 CSV



禁止：

- 把临时 OSS URL 保存为内容值。
- 为客户端 API 增加写路由。
- 擅自增加病毒扫描等 V1 非目标能力。

### 5.4 Agent 4：React 管理端

允许拥有：

```text
web/**
```

禁止：

- 直接修改 `internal/**`、`cmd/**`、`db/**`、`api/openapi/**` 和 `go.mod`。
- 自行改变 API 路由、请求字段和响应字段。
- 自行发明权限代码或字段类型名称。
- 将隐藏按钮作为权限安全边界。
- 将 API Key、OIDC Token 或会话数据保存到不安全的浏览器存储。
- 向后端提交未经约定处理的任意 HTML 富文本。

### 5.5 集成 Agent

允许拥有：

```text
cmd/cms/**
internal/app/**
db/migrations/**
api/openapi/admin.yaml
api/openapi/content.yaml
tests/integration/**
tests/e2e/**
Dockerfile
go.mod
go.sum
README.md
```

如果前端依赖文件位于 `web/`，则 `web/package.json` 和前端锁文件仍由 Agent 4 管理。

集成 Agent负责：

- 维护共享契约决议和冻结版本。
- 创建仓库骨架。
- 分配数据库迁移编号。
- 装配应用入口和模块。
- 注册路由。
- 聚合和验证 OpenAPI。
- 维护 Dockerfile。
- 连接 Go 和 React 构建。
- 维护集成测试和 E2E。
- 控制合并顺序和阶段发布。

集成 Agent不得：

- 接管功能模块的大量业务实现。
- 在集成 PR 中顺手重写工作流、权限或查询逻辑。
- 通过跨模块直接查询数据表打通 E2E。
- 静默改变已冻结契约。
- 重写其他 Agent 已合入的迁移。

## 6. 数据库迁移规则

`db/migrations/**` 是高冲突区域，按以下规则协作：

1. 业务 Agent提交迁移需求和 SQL 草案。
2. 集成 Agent预先分配唯一迁移编号。
3. 业务 Agent只在获分配的新文件中完善本领域 SQL。
4. 已合入或可能执行过的迁移不得修改。
5. 修正只能添加新的向前兼容迁移。
6. `schema_migrations` 和迁移执行器由集成 Agent独占。
7. 跨模块外键必须由双方和集成 Agent共同评审。
8. 避免跨模块外键形成迁移环和部署顺序耦合。

## 7. 契约冻结点

共享契约采用分层冻结，不在项目开始时一次性冻结全部细节。

### 7.1 F0：业务实现前

冻结：

- 仓库目录结构。
- Go 和前端依赖版本策略。
- ID 的 API 表示方式。
- UTC 和 RFC 3339 时间规范。
- JSON 命名规则。
- 管理 API 和内容 API 路径前缀。
- 统一错误结构。
- 请求 ID 传递方式。
- Handler、Service、Repository 边界。
- 数据库事务接口。
- 迁移编号和所有权规则。
- OpenAPI 分片和聚合规则。
- 配置注入方式。
- 自动化验证命令。

### 7.2 F1：阶段一并行实现前

冻结：

- `Principal` 结构。
- 会话失效语义。
- 系统权限和模型操作权限枚举。
- 审计事件最小字段和脱敏规则。
- 模型 DTO。
- 全部字段类型名称。
- 字段属性和默认值表达。
- 对象与重复组嵌套表达。
- 校验错误结构。
- 草稿内容请求和响应。
- 模型与字段归档语义。
- Revision 基础语义。
- React API Client 的消费形式。

阶段一验收后，已交付的管理 API V1 只能做兼容性扩展，不能进行破坏性修改。

### 7.3 F2：阶段二实现前

冻结：

- 工作流状态和合法转换。
- Revision、提交版本、草稿指针和发布指针语义。
- 并发审核原则。
- 投影类型编码。
- 动态唯一约束语义。
- 关联存储和展开语义。
- 过滤、排序和游标参数。
- 已发布内容读取服务接口。
- API Key 定位、验证和模型范围接口。
- ETag 生成依据。
- 内容 API 的错误码。

动态唯一字段按以下语义实施：

- 只约束未归档内容。
- 历史 Revision 不重复占用唯一值。
- 缺失值和 `null` 不参与唯一约束。
- 空字符串作为实际值参与唯一约束。
- 字符串使用明确且固定的 MySQL Collation。
- 小数标准化后投影。
- 日期时间转换为 UTC 后投影。

阶段二验收后，`/api/content/v1` 的结构、分页和 ETag 语义不得在阶段三改变。

### 7.4 F3：阶段三实现前

冻结：

- 素材 ID 在内容 JSON 中的表达。
- 素材状态。
- 上传申请和确认 DTO。
- OSS Adapter 接口。
- 素材引用维护和归档保护接口。
- CSV 列编码。
- 对象和重复组的 JSON 字符串格式。
- 导入错误结构。
- 后台任务状态、租约和重试语义。
- 导出筛选快照。
- 导出文件过期和下载权限。
- OSS 故障的可重试错误表示。

### 7.5 契约冻结产物

每次冻结至少包含：

1. OpenAPI 分片。
2. Go 公开接口或 DTO。
3. TypeScript 类型或消费示例。
4. 数据表责任清单。
5. 至少一个契约测试。
6. 兼容性规则。
7. 集成 Agent记录的冻结版本。

## 8. 阶段零：仓库骨架和共享契约

阶段零由集成 Agent顺序执行，预计 2 至 4 天。它用于解除并行开发阻塞，不增加产品功能。

### 8.1 Z0-I：集成骨架

任务：

- 提交设计和开发计划作为基线。
- 初始化 Go Module 并固定依赖版本。
- 初始化 React、TypeScript、Vite 和 Ant Design。
- 创建后端模块目录。
- 创建 MySQL 迁移执行框架。
- 创建应用配置入口。
- 建立 OpenAPI 3.1 管理 API和内容 API 聚合结构。
- 创建 Docker 多阶段构建骨架。
- 创建测试命令和测试 MySQL 8.0 约定。
- 创建统一错误结构和请求 ID。
- 创建模块注册接口，不注册虚假功能。
- 完成 F0 契约。

建议技术基线：

- Go 1.26。
- Node.js 24 LTS。
- React 19。
- TypeScript 5。
- MySQL 8.0。
- OpenAPI 3.1。

实际依赖版本通过 `go.sum` 和前端锁文件固定。

### 8.2 阶段零验收

```bash
go test ./...
go test -race ./...
go vet ./...
npm ci
npm run lint
npm run typecheck
npm run test
npm run build
docker build -t internal-cms:test .
docker run --rm internal-cms:test version
```

还需验证：

- MySQL 8.0 空库迁移成功。
- 重复执行迁移不会重复创建对象。
- 迁移状态异常时生产启动失败。
- 两份 OpenAPI 文档可以解析。
- Go 服务可以提供 React 静态资源。
- 配置和凭证没有进入前端构建产物。
- 未实现路由返回明确的 `404`。

## 9. 阶段一：基础平台与管理闭环

目标：系统可以部署，管理员可以登录、授权、创建动态模型并录入草稿。

### 9.1 第一并行波次

#### P1-A1：平台、身份、认证和会话

负责人：Agent 1。

任务：

- 环境变量配置和生产启动检查框架。
- MySQL 连接和事务基础。
- JSON 日志、请求 ID、请求耗时和操作者上下文。
- OIDC Authorization Code + PKCE。
- 校验 `state`、`nonce`、签名、issuer 和 audience。
- `(issuer, sub)` 本地用户映射。
- 服务端会话 Cookie。
- CSRF Token 和写请求同源检查。
- 空闲超时、绝对过期和禁用用户即时失效。
- 本地应急管理员登录。
- Argon2id 密码哈希。
- 创建和重置应急管理员命令的业务逻辑。
- 登录成功、失败、退出和密码重置审计。

完成标准：

- OIDC 测试使用受控测试发行方或 Fake，不依赖真实企业环境。
- Cookie 安全属性和 CSRF 均有测试。
- 不能删除或禁用最后一个应急管理员。
- 敏感数据不进入日志和审计。

#### P1-B1：动态模型和字段校验

负责人：Agent 2。

任务：

- 实现 `content_models` 和 `content_fields`。
- 实现全部 V1 字段类型和属性。
- 实现模型与字段管理 API。
- 实现模型和字段归档。
- 禁止修改稳定标识。
- 已有内容后禁止修改字段类型。
- 限制对象和重复字段组最多嵌套两层。
- 限制唯一、筛选和排序只能用于根级标量。
- 实现富文本结构化 JSON 校验。
- 实现稳定的校验错误码。
- 提供管理端和 CSV 可复用的模型 DTO。

完成标准：

- 每种字段类型覆盖合法、非法和边界测试。
- 标识修改、字段类型修改、嵌套超限和非法索引声明有测试。
- 未授权请求被拒绝。
- 模型变更产生审计事件。

#### P1-D1：React 应用壳、认证和权限导航

负责人：Agent 4。

先基于 OpenAPI Mock 开发：

- 登录页。
- SSO 发起和回调状态页。
- 本地应急登录入口。
- 当前用户信息。
- 会话失效处理。
- CSRF 请求处理。
- 统一 API Client。
- 管理端布局、导航和路由。
- 按权限隐藏入口和按钮。
- 统一错误展示。
- 桌面、平板和手机基础适配。

### 9.2 F1 冻结

第一波结束前，由集成 Agent聚合 Agent 1、Agent 2 和 Agent 4 的消费需求并冻结 F1。F1 未冻结前，不开始动态内容表单和草稿 API 的大规模实现。

### 9.3 第二并行波次

#### P1-A2：用户、角色、RBAC 和基础审计

负责人：Agent 1。

任务：

- 用户、角色和用户角色绑定。
- 系统权限和模型级权限。
- 多角色权限并集。
- 默认拒绝。
- 用户禁用。
- 用户、角色和权限管理 API。
- 只追加审计存储。
- 审计查询 API。
- 敏感字段过滤。

#### P1-B2：草稿内容 CRUD

负责人：Agent 2。

任务：

- 实现 `content_entries` 和 `content_revisions`。
- 每次有效保存创建不可变 Revision。
- 维护当前草稿版本指针。
- 实现草稿创建、查看、修改和归档。
- 执行动态模型服务端校验。
- 实现基础列表和分页。
- 执行权限校验和内容审计。

关键约束：

- 阶段一即采用 Revision，禁止在 `content_entries` 中原地覆盖内容 JSON。
- 保存 Revision 与更新草稿指针必须位于同一事务。
- 校验失败不能产生 Revision。

真实验收链路：


## 17. Agent 数量调整

### 17.1 只有 2 个实现 Agent

- Agent A 负责全部后端，按 Agent 1、Agent 2、Agent 3 的依赖顺序推进。
- Agent B 负责 React 管理端。
- 保留独立集成 Agent，或由 Agent A 在每阶段末承担集成工作。
- 预计周期增加到 10 至 14 周。

### 17.2 只有 3 个实现 Agent

- Agent A：平台、安全、认证、权限和审计。
- Agent B：模型、内容、工作流和 CSV。
- Agent C：API Key、内容 API、OSS 和 React。

该方案会使 Agent C 同时承担前后端，阶段二和阶段三压力集中，不作为首选。

### 17.3 共享同一工作目录

本计划假设每个 Agent 使用独立分支或 Git worktree。如果所有 Agent 共享同一个工作目录，则不能安全地并行写入，只能按任务包依赖顺序串行执行。

## 18. 关键风险

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| 阶段一原地更新内容 | 阶段二引入 Revision 时大规模返工 | 阶段一即创建不可变 Revision |
| 客户端 API 直接查询内容表 | 草稿泄露、发布语义分叉 | 只调用已发布内容读取接口 |
| 审计事务边界不清 | 审计与业务状态不一致 | F1 冻结成功和失败事件写入方式 |
| 前后端维护两套模型定义 | 字段类型和校验漂移 | OpenAPI 和模型 DTO 作为唯一契约 |
| 唯一字段语义未冻结 | 数据冲突和迁移返工 | F2 固定空值、归档、Collation 等语义 |
| 并发审核 | 发布指针覆盖或重复审核 | 条件更新或锁定，并执行并发测试 |
| 权限代码漂移 | 越权或功能不可用 | Agent 1 独占权限枚举，其他模块引用 |
| 多 Agent 修改迁移 | 编号和依赖冲突 | 集成 Agent统一分配迁移编号 |
| 多 Agent 修改 OpenAPI 根文件 | 高频合并冲突 | 业务 Agent维护分片，集成 Agent维护根文件 |
| 素材引用所有权不清 | 内容和素材状态不一致 | Agent 3 拥有表，Agent 2 通过契约维护引用 |
| CSV 流式与整批成功冲突 | 内存膨胀或半成功 | 临时表校验后再写正式内容 |
| Worker 依赖进程内状态 | 重启丢失或重复执行 | MySQL 持久任务、行锁和租约 |
| 未做代表性数据压测 | 上线后查询退化 | 阶段二开始执行百万级查询验证 |

## 19. 最终执行顺序

```text
建立基线提交
  -> 阶段零骨架和 F0
  -> F1
  -> 阶段一并行开发
  -> 阶段一集成验收
  -> F2
  -> 阶段二并行开发
  -> 阶段二集成验收
  -> F3
  -> 阶段三并行开发

```

## 20. 计划假设

本计划假设：

- 可以并行运行至少 5 个 Agent。
- 每个 Agent 可以使用独立分支或 Git worktree。
- MySQL 8.0 可以用于本地集成测试。
- 企业身份源支持标准 OIDC。
- 阿里云 OSS 测试环境可以在阶段三集成前提供。

如果 OIDC、OSS 或 MySQL 条件不成立，应先调整对应 Adapter 和集成测试方案，不应让业务 Agent自行加入未经评审的替代实现。
