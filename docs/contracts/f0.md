# F0 共享契约

- 冻结版本：F0
- 状态：已冻结
- 适用范围：管理 API、内容 API、Go 后端模块、数据库迁移和运行配置
- 依据：`docs/cms-design.md`、`docs/development-plan.md`
- 变更方式：共享契约变更须由集成 Agent 评审并产生新的冻结决议，不得由业务实现静默修改

F0 只冻结业务实现前所需的基础契约，不定义或承诺尚未实现的业务 API、DTO、权限枚举或业务错误码。

## 1. 基础表示

### 1.1 ID

- 所有 API 资源 ID 均表示为 JSON 字符串，OpenAPI schema 使用 `type: string`。
- ID 是不透明标识；调用方不得解析其数值、时间、分片或资源类型含义。
- ID 不得以 JSON number 输出，也不得依赖数据库自增值的大小或顺序。
- 同一资源的 ID 创建后保持稳定。

### 1.2 时间

- 数据库时间统一按 UTC 存储，应用内部时间在持久化和比较前转换为 UTC。
- API 日期时间使用 RFC 3339 字符串，OpenAPI schema 使用 `type: string` 和 `format: date-time`。
- API 输出统一使用 UTC，后缀为 `Z`；输入可接受 RFC 3339 中合法的时区偏移，并在进入业务逻辑后转换为 UTC。
- 时间精度不得超过微秒；调用方不得依赖未在具体契约中声明的精度。
- 仅日期值使用 RFC 3339 full-date，即 `YYYY-MM-DD`，不得附带隐式本地时区。

### 1.3 JSON 命名

- JSON 属性名、查询参数和稳定错误码统一使用小写 `snake_case`。
- 已发布字段名不得仅为统一风格而重命名；重命名属于破坏性变更。
- HTTP 标准头及本契约明确的 `X-Request-ID` 保持标准拼写，不受 `snake_case` 约束。

## 2. API 边界

- 管理 API 路径前缀固定为 `/api/admin/v1`。
- 内容 API 路径前缀固定为 `/api/content/v1`。
- 两类 API 分别维护 OpenAPI 3.1 根文档，不共享业务路由。
- 未实现功能不得注册运行时路由，也不得以成功响应提供占位实现；未注册路径应明确返回 `404`。

## 3. 统一错误

所有由应用生成的非成功 API 响应使用 `api/openapi/fragments/shared/error.yaml` 中的 `ErrorResponse`：

```json
{
  "error": {
    "code": "validation_failed",
    "message": "请求数据校验失败",
    "request_id": "req_xxx",
    "details": []
  }
}
```

- `code` 是稳定的、非空的 `snake_case` 机器可读错误码；调用方只能依赖该字段判断错误类型。
- `message` 是非空的人类可读说明，可以随文案和语言调整，不是稳定判断依据。
- `request_id` 必须与当前响应的 `X-Request-ID` 完全一致。
- `details` 始终存在且为数组；无详情时返回空数组。元素的具体字段推迟至对应业务冻结点定义。
- 错误响应不得泄露堆栈、SQL、凭证、Token、Session、完整 API Key、OSS 签名 URL 或其他敏感信息。
- 代理或基础设施在请求到达应用前生成的响应不受应用错误体约束，但部署层应尽可能保留请求 ID。

## 4. 请求 ID

- 请求 ID 使用 HTTP 请求头和响应头 `X-Request-ID` 传递。
- 入口收到非空且符合实现长度与字符安全限制的 `X-Request-ID` 时沿用；缺失或不合法时生成新的不透明请求 ID。
- 应用对每个已接收请求返回 `X-Request-ID`，并在错误体、结构化日志和相关审计事件中使用同一值。
- 请求 ID 仅用于关联和诊断，不作为认证凭据、授权依据、幂等键或资源 ID。
- 下游调用应传递当前请求 ID；后台任务没有入口请求时生成新的请求 ID。

## 5. 分层职责

### Handler

- 负责 HTTP 路由适配、认证上下文读取、请求解析、基础格式校验、调用 Service，以及将结果映射为 HTTP 响应。
- 不承载业务规则，不直接执行 SQL，不直接访问 Repository，不直接修改任何模块的数据表。
- 不自行开启跨 Service 的隐式事务。

### Service

- 负责业务规则、权限调用、模块间公开接口调用、事务边界和领域错误。
- 一个模块只通过其他模块公开的 Service 接口协作，不直接访问其他模块 Repository 或数据表。
- 需要原子一致性的业务写入由拥有该用例的 Service 在一个显式事务中编排。

### Repository

- 负责本模块数据的显式 SQL 和持久化映射，不包含 HTTP、认证或业务流程判断。
- 方法接收显式查询或事务执行上下文，不自行提交由 Service 发起的事务。
- Repository 不得访问其他模块拥有的数据表；跨模块数据通过公开 Service 接口取得。

依赖方向固定为 `Handler -> Service -> Repository`。允许 Service 调用其他模块公开 Service 接口，不允许反向依赖或跨层捷径。

## 6. 事务边界

- 事务由 Service 层开启、提交和回滚；Handler 与 Repository 不拥有业务事务边界。
- 事务回调或事务句柄必须显式传递，禁止依赖进程全局或上下文中的隐藏可变事务。
- 同一业务不变量涉及的数据库写入必须处于同一 MySQL 事务，包括业务状态及其要求原子写入的投影、关系、引用或成功审计事件。
- 事务内的 Repository 必须使用同一事务执行器，不得退回普通连接造成部分提交。
- 任一步骤失败必须回滚；提交失败按整体失败处理，不得向调用方报告成功。
- 不在数据库事务内持有不可控的外部网络调用。需要跨 MySQL 与外部系统协调时，由后续阶段明确状态机或补偿语义，不声称分布式原子性。
- 失败尝试审计若需在业务回滚后保留，必须使用后续审计契约明确的独立写入语义，不得与已回滚业务状态混淆。

## 7. 迁移

- 迁移文件位于 `db/migrations/**`，唯一编号由集成 Agent 预先分配。
- 编号采用固定宽度六位十进制、全局单调递增格式，例如 `000001_<snake_case_name>.up.sql`；同一编号不得复用。
- 名称使用小写 `snake_case`，应描述结构变更而非 Agent 或临时任务名。
- 业务 Agent 只可完善获分配的新迁移文件；`schema_migrations` 和迁移执行器由集成 Agent 独占。
- 已合入或可能执行过的迁移不可修改、重排、删除或复用；修正只能增加更高编号的向前兼容迁移。
- 每个迁移文件只包含一个可独立验证的 DDL 或 DML 语句；MySQL DDL 不具备跨语句事务原子性，复杂变更必须拆成多个迁移并逐步预检。
- 已执行迁移必须是当前二进制迁移列表的连续前缀，禁止跳过前置版本后倒序补跑。
- 跨模块外键须经双方与集成 Agent 共同评审，并避免形成迁移环和部署顺序耦合。
- 生产启动必须校验迁移状态；存在未知版本、校验异常或未满足的迁移前置条件时启动失败。

## 8. 模块注册

- 应用入口由集成层显式创建依赖并注册模块，业务包不得通过 `init()`、全局副作用或导入即注册的方式接入运行时。
- 每个模块通过公开注册入口接收已构造的配置、基础设施和跨模块 Service 接口，并向应用路由器注册本模块已实现的 Handler。
- 注册顺序由应用装配层明确控制；模块不得依赖偶然的包导入或 map 遍历顺序。
- 模块注册不得直接读取环境变量、创建未声明的单例或绕过依赖注入服务定位器。
- 未实现的模块或功能不注册路由，不提供返回成功的空实现。
- 应用入口和跨模块装配由集成 Agent 维护，业务模块不能直接修改应用入口以接入自身。

## 9. OpenAPI 分片

- `api/openapi/admin.yaml` 与 `api/openapi/content.yaml` 是 OpenAPI 3.1 根文档，仅由集成 Agent 维护。
- 管理业务分片位于 `api/openapi/fragments/admin/<domain>/**`；内容业务分片位于 `api/openapi/fragments/content/**`；共享 schema 位于 `api/openapi/fragments/shared/**`。
- 业务 Agent 只维护其目录所有权内的分片，不直接修改根文档。
- 分片使用相对 `$ref`，引用目标必须可解析；共享定义只有在管理 API 与内容 API 语义完全一致时才能复用。
- 集成 Agent 负责将已实现、已测试且处于当前阶段范围内的分片聚合进根文档，并验证两份文档可独立解析。
- OpenAPI 中的路由、方法、请求和响应必须与运行时一致。不得描述尚未实现的业务 API，也不得注册未进入 OpenAPI 的已发布业务路由。
- 当前两个根文档的 `paths` 保持为空；后续仅在对应业务实现完成并达到冻结点后加入路径。

## 10. 配置注入

- 运行配置仅在应用入口从环境变量读取、解析和校验一次，再以显式、不可变的类型化配置注入所需模块。
- 业务模块、Handler、Service 和 Repository 不直接调用环境变量读取函数，不依赖进程全局配置。
- 必需配置缺失、格式非法或生产安全检查失败时，应用在注册路由和启动服务前快速失败。
- 配置项名称以 `docs/cms-design.md` 第 18 节为基线；新增配置须经集成评审并记录装配变化。
- 密钥和凭证不得设置仓库内生产默认值，不得写入镜像、日志、审计、错误响应或前端构建产物。
- 测试通过显式构造配置覆盖依赖，不修改全局环境作为业务单元测试的隐式输入。

## 11. 兼容性规则

- F0 冻结后，ID 的字符串表示、时间格式、JSON 命名、路径前缀、统一错误外壳和请求 ID 传递方式均视为稳定契约。
- 兼容性扩展包括：新增可选字段、新增业务路由、新增 HTTP 成功或错误响应，以及放宽输入约束；前提是旧调用方行为保持有效且语义不变。
- 破坏性变更包括：删除或重命名字段、改变字段类型或含义、将可选字段改为必填、收紧既有输入、修改路径或方法、移除既有状态码或错误码、改变事务原子性承诺。
- 新增必填响应字段只有在旧客户端可安全忽略未知字段且对应 schema 允许时才可作为兼容扩展；新增必填请求字段始终是破坏性变更。
- 错误 `message` 文案可调整；稳定 `code` 的含义不得改变或复用于其他错误。
- 破坏性 API 变更必须使用新的 URL 主版本并形成新的冻结决议，不能在 `/v1` 中静默发布。
- OpenAPI、实现、测试和消费方示例必须在同一变更中保持一致；如发生冲突，以经评审的冻结决议为准，不以当前偶然实现为准。
- 数据库演进采用扩展优先和向前兼容迁移，应用回滚不得依赖回滚或重写已经执行的数据迁移。

## 12. F0 自检基线

- 两份 OpenAPI 根文档均为 `3.1.0`，可以独立解析，且 `paths` 为空。
- 两份根文档均通过相对 `$ref` 引用同一个 `ErrorResponse` schema。
- OpenAPI 文件不包含任何尚未实现的业务 API。
- 后续任务包须检查目录所有权、分片可聚合性、错误结构、跨层调用、事务边界和迁移编号。
- 自动化门禁沿用开发计划：`go test ./...`、`go test -race ./...`、`go vet ./...`、前端 lint/typecheck/test/build 及容器构建；仅在对应骨架存在后执行。
