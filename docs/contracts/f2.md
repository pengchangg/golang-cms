# F2 冻结契约

- 冻结版本：F2
- 状态：已冻结并集成
- 适用范围：阶段二工作流、投影与关联、已发布内容读取、API Key 和客户端只读 API
- 依据：`docs/cms-design.md`、`docs/development-plan.md`、`docs/contracts/f0.md`、`docs/contracts/f1.md` 和当前阶段一实现
- OpenAPI 分片：`api/openapi/fragments/admin/content/**`、`admin/api-keys/**`、`content/**`
- 集成状态：本文新增的 OpenAPI 操作均已标记 `x-integration-status: integrated` 并聚合进根 OpenAPI

F2 只做 F1 的兼容扩展。F1 已冻结的条目生命周期、不可变 Revision、草稿指针、唯一值占用和管理 API 保持有效。本文不注册运行时路由，不修改根 OpenAPI、Go 代码或迁移。

## 1. 状态归属和工作流

五个工作流状态固定为：

```text
draft
pending_review
rejected
published
unpublished
```

工作流状态属于 Revision，不替代 F1 的条目生命周期 `draft/archived`。Revision 的 `content`、`number`、`created_by` 和 `created_at` 不可变；工作流状态和审核元数据由条件转换更新，每次转换另写不可变工作流事件和审计事件。

合法转换固定如下：

| 当前状态 | 动作 | 目标状态 | 条件 |
| --- | --- | --- | --- |
| `draft` | `submit` | `pending_review` | 操作者有 `content.submit`，目标是当前草稿指针 |
| `pending_review` | `approve` | `published` | 操作者同时有 `content.review`、`content.publish`，且不是提交人 |
| `pending_review` | `reject` | `rejected` | 操作者有 `content.review`，且不是提交人，理由有效 |
| `rejected` | `edit` | 新 Revision 为 `draft` | 操作者有 `content.update`，以被驳回 Revision 为基线 |
| `published` | `edit` | 新 Revision 为 `draft` | 操作者有 `content.update`，线上 Revision 不变 |
| `published` | `unpublish` | `unpublished` | 操作者有 `content.unpublish`，目标是当前发布指针 |
| `unpublished` | `edit` | 新 Revision 为 `draft` | 操作者有 `content.update`，以当前作者工作头为基线 |

除表中转换外全部非法。特别是：

- `pending_review` 不可编辑、重复提交、重复通过或重复驳回。
- `rejected` 不可直接重新提交、通过或发布，必须先保存一个新 `draft` Revision。
- `unpublished` 不可直接重新发布，必须经过编辑、提交和审核；一次显式保存即使内容相同也创建新 Revision。
- 不提供撤回审核、跳过审核发布、回滚 Revision 或恢复归档条目的动作。
- 条目归档不是工作流转换。存在当前发布指针的条目不得归档，返回 `409 published_entry_archive_forbidden`；应先下线。

## 2. 提交、审核和指针

### 2.1 提交人和自审

- `submitted_by` 在 `draft -> pending_review` 时固定为当前用户，后续不可修改。
- `submitted_at` 同时固定为当前 UTC 时间。
- 禁止提交人执行该 Revision 的 `approve` 和 `reject`。比较稳定用户 ID，不比较角色、显示名或当前权限。
- 自审返回 `409 self_review_forbidden`，不得改变状态、指针或工作流事件。
- 提交后用户被禁用或权限变化不改变已记录的提交人；审核操作者仍按请求时有效权限判断。

### 2.2 驳回理由

- `reject` 请求必须包含 `reason`，去除首尾 Unicode 空白后长度为 1 至 1000 个 Unicode 码点。
- 服务端保存去除首尾空白后的值。只含空白、超长或类型错误返回 `400 validation_failed`，详情路径 `/reason`。
- 驳回理由属于工作流事件，不写入 Revision 内容；管理端版本历史可见，客户端 API 永不返回。

### 2.3 草稿指针

- F1 的 `current_draft_revision_id` 数据库指针继续表示当前作者工作头，不承诺其目标 Revision 的工作流状态一定为 `draft`。
- 创建和编辑后，它指向新 `draft` Revision；提交、通过、驳回和下线不移动该指针。
- `pending_review` 工作头不可编辑。`rejected`、`published` 或 `unpublished` 工作头被编辑时创建新 Revision，并原子切换草稿指针。
- 条目存在一个较新的草稿且仍有旧发布版本时，草稿指针和发布指针分别指向不同 Revision。

### 2.4 发布指针

- 每个条目最多一个 `current_published_revision_id`。从未发布或已下线时为 `null`。
- 审核通过必须在同一事务中完成：锁定模型和条目、条件转换提交 Revision、把旧发布 Revision 从 `published` 改为 `unpublished`（如有）、把目标 Revision 改为 `published`、切换发布指针、写工作流事件和成功审计。
- 发布新 Revision 不移动草稿指针。编辑已发布条目不会改变发布指针，客户端继续读取旧发布 Revision。
- 下线必须在同一事务中把当前发布 Revision 改为 `unpublished`、删除发布指针并写事件和审计。事务提交后客户端详情返回 `404 published_content_not_found`，列表不再包含该条目。
- 发布指针是客户端可见性的唯一依据。不得仅凭 Revision 状态或条目更新时间判断已发布。

## 3. 版本级并发条件

- 所有工作流请求显式携带目标 `revision_id`。`submit`、`approve`、`reject` 要求它等于当前草稿指针；`unpublish` 要求它等于当前发布指针。
- 每个动作在事务内按 `model -> entry -> revision/pointer` 顺序加锁，并以目标 Revision 的预期前置状态执行条件更新。
- 指针不匹配返回 `409 workflow_revision_conflict`；状态不匹配返回 `409 invalid_workflow_transition`。
- 两个并发审核、审核与驳回、审核与草稿变更中最多一个成功。失败请求不得覆盖成功结果，也不得追加成功工作流事件或成功审计。
- 编辑仍使用 F1 的 `base_revision_id`。它必须等于草稿指针，且该 Revision 状态允许编辑；基线不匹配仍返回 `409 draft_revision_conflict`。
- 不使用条目级 `updated_at` 代替版本条件，客户端也不得通过先读后写推断动作一定成功。

## 4. 管理工作流 API

新增路由固定为：

```text
POST /api/admin/v1/models/{model_id}/entries/{entry_id}/submit
POST /api/admin/v1/models/{model_id}/entries/{entry_id}/approve
POST /api/admin/v1/models/{model_id}/entries/{entry_id}/reject
POST /api/admin/v1/models/{model_id}/entries/{entry_id}/unpublish
GET  /api/admin/v1/models/{model_id}/entries/{entry_id}/workflow-events
```

- 四个动作请求都含必填 `revision_id`；驳回额外含必填 `reason`。
- 动作成功返回更新后的 `WorkflowEntry`。工作流事件列表使用 `{items,next_cursor}`，按 `occurred_at DESC,id DESC`。
- `WorkflowEntry` 在 F1 `ContentEntry` 基础上增加 `workflow_status`、`current_published_revision_id`、`current_published_revision`；发布指针和发布 Revision 可为 `null`。
- 管理条目列表兼容增加 `workflow_status`、`filter`、`relation_filter`、`sort`、`expand` 和 `include_total`。未传时保留 F1 行为和排序。
- `include_total=true` 时响应增加 `total`，最大计数 10,000；实际匹配超过上限时 `total=10000`、`total_is_estimate=true`。默认不计数，两个字段均省略。

## 5. 关联内容值和存储

关联字段内容值固定为：

| 字段类型 | JSON 表示 |
| --- | --- |
| `single_relation` | 目标内容条目 ID string，缺省或 `null` 表示无关联 |
| `multi_relation` | 目标内容条目 ID 的 array，顺序有意义，不允许重复，最多 50 项 |

- 目标条目必须属于字段 `target_model_id`，存在且未归档。保存时不要求目标已发布。
- 关联只允许根级字段；F2 不放宽对象或重复组中的关联持久化。
- 每个 Revision 的关联行独立保存，至少包含 `revision_id`、源 `entry_id/model_id`、`field_id`、目标 `entry_id/model_id` 和从 0 开始的 `position`。
- 单关联固定 `position=0`；多关联按 JSON 数组顺序存储。唯一约束防止同一 Revision、字段、目标重复，位置约束保证顺序唯一。
- Revision JSON、投影行和关联行在创建 Revision 的同一事务写入。历史 Revision 的关联行不可更新或删除。
- 目标条目之后归档或下线不改写源 Revision。管理端读取仍返回原始 ID；客户端展开时按当前发布指针决定是否返回展开对象。

## 6. 类型化投影

`content_field_values` 是 Revision 级投影，每个 Revision 的每个已声明 `unique`、`filterable` 或 `sortable` 的根级标量非 `null` 字段恰好一行。投影不替代 Revision JSON。

类型标签和列固定如下：

| 字段类型 | `value_type` | 值列 | 规范化 |
| --- | --- | --- | --- |
| 文本、单选 | `string` | `string_value` | UTF-8 原字符串，不裁剪、不折叠大小写，`utf8mb4_0900_as_cs` |
| 整数 | `integer` | `integer_value` | 有符号 64 位十进制整数 |
| 小数 | `decimal` | `decimal_value` | `DECIMAL(65,30)`；负零为零，比较忽略末尾零 |
| 布尔 | `boolean` | `boolean_value` | `0` 或 `1` |
| 日期 | `date` | `date_value` | RFC 3339 full-date |
| 日期时间 | `datetime` | `datetime_value` | UTC `DATETIME(6)`；超过微秒的输入按 F0 拒绝 |

- 一行只能有与 `value_type` 对应的一个非空值列，其余值列必须为 `null`。
- 缺失值和 JSON `null` 不写投影。空字符串写入并参与过滤、排序和唯一性。
- 投影索引以 `(model_id,field_id,value)` 开头，并包含 `revision_id`、`entry_id` 以支持指针连接和稳定分页。
- 对声明投影能力的整数和小数字段，保存值必须落入对应投影范围；超出返回 `400 validation_failed`，详情代码 `projection_out_of_range`。
- 常用投影查询不得依赖 `JSON_EXTRACT` 全表扫描。

## 7. 动态唯一字段和现有数据演进

F1 的 `content_unique_values` 从“每条目每字段一个当前草稿值”兼容演进为“当前草稿指针与当前发布指针的值并集”占用表：

- 只约束未归档条目。一个条目存在不同的当前草稿值和当前发布值时，两者都占用；值相同时只占一行。
- 不被草稿指针或发布指针引用的历史 Revision 不占用。编辑、发布、下线和归档都必须在指针变化的同一事务内重算该条目的占用并集。
- 编辑已发布内容不会提前释放仍在线的旧发布值；新版本发布后，旧值只有在不再被草稿或发布指针引用时才释放。
- 缺失值和 `null` 不参与；空字符串参与。
- canonical 输入继续使用 F1 类型标签、NUL 分隔符和规范值，数据库保存 SHA-256；字符串唯一性按原始 UTF-8 字节判定。`utf8mb4_0900_as_cs` 固定用于字符串投影过滤和排序，不反向改变 F1 已冻结的唯一值哈希。
- 唯一冲突仍返回 `400 validation_failed`，字段详情代码 `not_unique`。
- 现有唯一键 `(model_id,field_id,canonical_value)` 保持全模型裁决；必须移除或扩展 F1 的 `(entry_id,field_id)` 单值约束，使同一条目字段可占用两个不同值，并增加 `(entry_id,field_id,canonical_value)` 唯一约束防止重复行。

模型已有内容时，字段约束按以下方式演进：

- `unique/filterable/sortable` 从 `false` 改为 `true` 前，必须在同一受控操作中预检全部现有 Revision，并为投影表回填；启用 `unique` 还须按全部未归档条目的当前草稿指针与当前发布指针并集预检并回填唯一占用。
- 任一值无法编码或存在唯一冲突时，字段更新整体失败，不改变字段声明，不留下部分投影或占用。
- 从 `true` 改为 `false` 可以停止后续对应查询能力；删除派生投影应延后清理，不能删除 Revision JSON 或历史关联。
- 实现若不能在管理请求的资源限制内安全完成回填，必须返回 `409 field_projection_backfill_required`，由后续受控运维流程处理；不得先启用声明再异步暴露不完整查询。

## 8. 查询参数和限制

管理和客户端内容列表共用以下查询语义：

- `limit`：默认 20，最小 1，最大 100。
- `cursor`：不透明字符串，绑定 API 受众、模型、API Key 模型范围、规范化过滤、排序、展开和页大小。绑定项变化返回 `400 invalid_cursor`。
- `filter`：URL 编码 JSON object，最多 5 个字段。键是稳定字段 key，值是只含一个操作符的 object。
- `relation_filter`：URL 编码 JSON object，最多 2 个关联字段。每项固定为 `{"contains":"<entry_id>"}`。
- `sort`：逗号分隔，最多 3 个字段；前缀 `-` 表示降序，无前缀表示升序。例如 `-published_at,title`。
- `expand`：逗号分隔的关联字段 key，最多 3 个；仅允许根级关联字段，只展开一层。

标量操作符固定为：

| 类型 | 允许操作符 |
| --- | --- |
| 文本、单选、布尔 | `eq`、`ne`、`in` |
| 整数、小数、日期、日期时间 | `eq`、`ne`、`gt`、`gte`、`lt`、`lte`、`in` |

- `in` 的值为同类型 JSON array，1 至 20 项且不重复；其他操作符的值为字段 JSON 标量表示。
- 只允许过滤 `filterable=true` 字段，排序 `sortable=true` 字段。内建客户端排序字段 `published_at` 和 `id` 始终可用。
- 客户端默认排序 `-published_at,-id`。自定义排序末尾由服务端追加同方向 `id` 作为唯一稳定决胜键。
- 管理端未指定 `sort` 时继续使用 F1 的 `-updated_at,-id`；Revision 和工作流事件列表不接受动态过滤排序。
- 未知字段、未声明能力、错误类型、未知操作符、空排序项或超限统一返回 `400 invalid_query`；查询参数语法错误的 `details` 为空。
- 游标分页不提供跨请求快照隔离，但在底层发布集合未变化时不得重复或遗漏。

## 9. PublishedContentReader Go 语义

内容模块向客户端模块提供以下公开读取接口。此节冻结包边界、方法和行为，F2 实现必须据此增加 Go 类型；客户端 Handler 不得直接查询内容表。

```go
type PublishedContentReader interface {
	ListPublishedModels(ctx context.Context, allowedModelIDs []string) ([]PublishedModel, error)
	GetPublishedModel(ctx context.Context, modelKey string, allowedModelIDs []string) (PublishedModel, error)
	ListPublishedEntries(ctx context.Context, modelKey string, allowedModelIDs []string, query PublishedQuery) (PublishedEntryPage, error)
	GetPublishedEntry(ctx context.Context, modelKey, entryID string, allowedModelIDs []string, expand []string) (PublishedEntry, error)
}

type PublishedQuery struct {
	Limit          int
	Cursor         string
	Filters        []PublishedFilter
	RelationFilters []PublishedRelationFilter
	Sort           []PublishedSort
	Expand         []string
}
```

- `modelKey` 是路径中的稳定模型 key。Reader 在内容模块内部解析模型并同时应用范围，不向客户端模块暴露模型或内容表查询接口。
- `allowedModelIDs` 是客户端模块从已验证 API Key 得到并去重的授权范围；空 slice 表示无模型，不表示全部模型。Reader 对不存在、归档或不在范围内的 model key 返回同一个领域级 not found。模型列表只含 active 模型并按 `key ASC,id ASC`。
- Reader 只通过发布指针读取 Revision，不接受“包含草稿”开关。无发布指针的详情返回领域级 not found；列表静默排除。
- Reader 负责字段能力校验、投影和关联查询、游标正确性及一层展开；客户端模块负责 HTTP 解析、Bearer 鉴权、模型范围和 ETag。
- `PublishedEntry.Content` 保留原始关联 ID；展开结果单独放在 `expanded`，键为字段 key。未发布、已归档、越过模型范围或不存在的目标不出现在 `expanded`，且不递归展开。
- Reader 返回的切片和 `json.RawMessage` 视为调用方只读；实现不得返回依赖事务关闭后才可读取的对象。
- `context.Canceled` 和 `context.DeadlineExceeded` 原样传播；领域错误使用稳定分类，其他存储错误不得伪装为空列表或 not found。

## 10. API Key 契约

### 10.1 密钥格式和验证

- 完整密钥格式固定为 `cmsk_<prefix>_<secret>`。
- `prefix` 是 12 个小写 Base32 字符，用于唯一定位数据库记录，可记录到访问日志；`secret` 是 32 个随机字节的无填充 Base64URL 编码。
- 数据库只保存 prefix、每 Key 独立 16 字节随机 salt 和 `SHA-256(salt || secret_bytes)`。完整密钥和 secret 不得持久化、写日志、审计或错误。
- 解析格式、按 prefix 定位、检查状态后，使用常量时间比较验证哈希。未知 prefix、格式错误或哈希错误统一返回 `401 invalid_api_key`。
- Bearer 头固定为 `Authorization: Bearer <api-key>`。缺失返回 `401 api_key_required`，重复头或非 Bearer scheme 返回 `401 invalid_api_key`。

### 10.2 模型范围、过期和撤销

- 每个 Key 至少授权一个 active 模型，模型 ID 去重。空数组不表示全模型。
- Key 只能读取授权模型；模型列表只返回授权交集。访问范围外模型统一返回 `404 published_model_not_found`，避免泄露模型存在性。
- `expires_at` 可为 `null`；非空时创建时必须晚于当前时间。服务端在 `now >= expires_at` 时返回 `401 api_key_expired`。
- `revoked_at` 非空即立即失效，返回 `401 api_key_revoked`。重复撤销返回 `409 api_key_already_revoked`。
- `last_used_at` 可为 `null`，成功鉴权后异步或节流更新，最多每 5 分钟写一次；更新失败不改变本次已完成鉴权结果。

### 10.3 轮换

- 轮换请求可设置新 `name`、`model_ids` 和 `expires_at`；省略时继承旧 Key 当前值。
- 轮换在同一事务创建新 Key、设置 `rotated_from_id`、把旧 Key 设为 revoked 并设置 `replaced_by_id`。旧 Key 在事务提交后立即失效，不提供重叠宽限期。
- 轮换成功返回新 Key 元数据和仅出现一次的完整 `key`。响应不得缓存，使用 `Cache-Control: no-store`。
- 已撤销或已过期 Key 不可轮换，分别返回 `409 api_key_already_revoked`、`409 api_key_expired`。

## 11. API Key 管理路由

路由固定为：

```text
GET, POST /api/admin/v1/api-keys
GET        /api/admin/v1/api-keys/{api_key_id}
DELETE     /api/admin/v1/api-keys/{api_key_id}
POST       /api/admin/v1/api-keys/{api_key_id}/rotate
```

- 列表和详情要求 `api_keys.view`，创建和轮换要求 `api_keys.create`，撤销要求 `api_keys.revoke`。
- 创建请求包含 `name`、`model_ids`、可空 `expires_at`。名称去除首尾空白后 1 至 120 个码点。
- 创建和轮换返回 `APIKeySecret`，完整 `key` 只在该响应出现。列表和详情永不返回 hash、salt 或完整 key。
- 列表按 `created_at DESC,id DESC`，使用 F1 游标外壳；可按 `status=active|expired|revoked` 精确筛选。
- 创建、撤销和轮换分别审计 `api_key_created`、`api_key_revoked`、`api_key_rotated`，changes 只含 ID、prefix、模型范围和非敏感元数据。

## 12. 客户端只读 API

客户端路由固定为：

```text
GET /api/content/v1/models
GET /api/content/v1/models/{model_key}
GET /api/content/v1/models/{model_key}/entries
GET /api/content/v1/models/{model_key}/entries/{entry_id}
```

- 只注册 `GET`。同路径的 `POST`、`PUT`、`PATCH`、`DELETE` 不存在，不提供写入或预览能力。
- 模型路径使用稳定 `model_key`，不是内部 ID。响应仍同时返回不透明 `id` 和 `key`。
- 模型列表只返回 API Key 授权且 active 的模型；模型详情只返回 active 字段。归档模型对客户端视为不存在。
- 条目只读取当前发布指针。Revision 历史、工作流状态、提交人、审核人、驳回理由、草稿指针和内部审计字段不返回。
- `PublishedEntry` 固定含 `id`、`model_id`、`model_key`、`revision_id`、`revision_number`、`content`、`expanded`、`published_at` 和 `updated_at`。其中 `updated_at` 等于该发布 Revision 的创建时间，不受草稿编辑影响。
- `expand` 默认关闭，最多一层。原始 `content` 中关联 ID 始终保留；`expanded` 只包含成功展开的已发布目标。未发布、归档、不存在或超出 Key 模型范围的关联不展开，不把原因暴露给客户端。

## 13. ETag 和条件请求

- 四个客户端 GET 成功响应都返回强 ETag，格式为引号包裹的 Base64URL SHA-256，例如 `"sha256-..."`。
- ETag 输入是最终响应体的确定性 JSON 编码。对象键按 UTF-8 字节序、数组保持响应顺序、数字和字符串使用契约规范表示；不包含请求 ID、`last_used_at` 或传输编码。
- 列表 ETag 因条目集合、发布指针、排序、过滤、游标页或展开结果变化而变化。详情 ETag 因当前发布 Revision 或展开目标的当前发布 Revision 变化而变化。
- `If-None-Match` 按 RFC 9110 接受逗号分隔标签和 `*`，执行弱比较；任一匹配返回 `304`，无响应体，并返回当前 `ETag`、`X-Request-ID` 和缓存头。
- 鉴权、模型范围和查询校验先于条件判断。无效或越权 Key 不得因 ETag 匹配得到 `304`。
- 成功和 `304` 固定返回 `Cache-Control: private, no-cache`；API Key 创建和轮换响应固定 `no-store`。

## 14. 路由 DTO 和稳定错误码

统一错误外壳沿用 F0；动态请求体校验沿用 F1 `{path,code,message}` details。F2 新增稳定顶层错误码：

| HTTP | `error.code` | 适用情况 |
| --- | --- | --- |
| 400 | `validation_failed` | 请求体字段、驳回理由或模型 ID 数组校验失败 |
| 400 | `invalid_query` | 过滤、排序、展开或参数限制不合法 |
| 400 | `invalid_cursor` | 游标格式错误或绑定条件变化 |
| 401 | `api_key_required` | 内容 API 缺少 Authorization |
| 401 | `invalid_api_key` | 格式、prefix、secret 或 scheme 无效 |
| 401 | `api_key_expired` | Key 已过期 |
| 401 | `api_key_revoked` | Key 已撤销或已被轮换替代 |
| 403 | `permission_denied` | 管理用户缺少动作权限 |
| 404 | `resource_not_found` | 管理 API 的模型、条目、Revision 或 Key 不存在 |
| 404 | `published_model_not_found` | 客户端模型不存在、归档或不在 Key 范围 |
| 404 | `published_content_not_found` | 客户端条目无当前发布指针或不可见 |
| 409 | `draft_revision_conflict` | 编辑基线不是当前草稿指针 |
| 409 | `workflow_revision_conflict` | 动作目标不是要求的当前指针 |
| 409 | `invalid_workflow_transition` | Revision 不处于动作要求的前置状态 |
| 409 | `self_review_forbidden` | 提交人尝试审核自己的 Revision |
| 409 | `published_entry_archive_forbidden` | 归档仍在线的条目 |
| 409 | `field_projection_backfill_required` | 现有内容不能在当前请求安全完成能力回填 |
| 409 | `api_key_already_revoked` | 重复撤销或轮换已撤销 Key |
| 409 | `api_key_expired` | 管理端尝试轮换已过期 Key |

不存在、越权和不可见按表中语义映射，不通过不同 message 泄露内部状态。`304` 不是错误，不返回 `ErrorResponse`。

## 16. 实现与兼容门禁

- 全覆盖合法和非法状态转换、禁止自审、驳回理由、并发审核和发布指针原子性。
- 验证编辑已发布内容不改变客户端结果，新版本通过后才切换；下线提交后立即不可读。
- 验证 JSON、投影、关联、唯一占用、指针、工作流事件和成功审计的事务一致性。
- 验证类型编码边界、现有数据回填冲突、空值、空字符串、字符串 Collation、小数和 UTC 日期时间。
- 验证过滤和排序声明、数量限制、游标绑定、无深度 OFFSET、无常用 `JSON_EXTRACT` 全表扫描。
- 验证 API Key 正确、错误、过期、撤销、轮换、模型越权、一次性 secret 和日志审计脱敏。
- 验证内容 API 只读、只发布、一层展开、未发布关联不展开，以及 ETag 的 `200/304` 行为。
- F2 分片已完成实现和契约测试，标记为 `integrated` 并聚合根 OpenAPI。
- 阶段二验收后，`/api/content/v1` 的 DTO、查询、分页、展开和 ETag 语义只能兼容扩展，阶段三不得修改。
