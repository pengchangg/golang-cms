# F1 冻结契约

- 冻结版本：F1-wave-2
- 状态：已实现并通过阶段一集成验收
- 适用范围：第一波全部契约，以及用户、角色、授权、审计查询和草稿内容
- 依据：`docs/cms-design.md`、`docs/development-plan.md`、`docs/contracts/f0.md` 和当前 F0 平台骨架
- OpenAPI 分片：`api/openapi/fragments/admin/auth/**`、`identity/**`、`roles/**`、`audit/**`、`models/**`、`content/**`
- 集成状态：阶段一认证、用户、角色、审计、模型、字段、草稿内容和 Revision 路径均已聚合进 `api/openapi/admin.yaml`

当前认证仅支持短信登录和本地应急登录；历史 OIDC 路由不注册，历史 OIDC 会话在迁移中撤销并删除。第二波路径在实现前不得注册，也不得返回占位成功响应。

## 1. 认证路由和 DTO

完整运行时路由固定如下：

| 方法 | 路由 | 用途 | 成功响应 |
| --- | --- | --- | --- |
| `POST` | `/api/admin/v1/auth/captcha/challenges` | 创建滑动拼图挑战 | `201` `CaptchaChallengeResponse` |
| `POST` | `/api/admin/v1/auth/sms/challenges` | 创建短信登录挑战 | `201` `SMSChallengeResponse` |
| `POST` | `/api/admin/v1/auth/sms/challenges/{challenge_id}/verify` | 验证短信并建立会话 | `200` `SessionResponse` |
| `POST` | `/api/admin/v1/auth/local/login` | 本地应急管理员登录 | `200` `SessionResponse` |
| `GET` | `/api/admin/v1/auth/session` | 读取当前会话、Principal 和 CSRF Token | `200` `SessionResponse` |
| `POST` | `/api/admin/v1/auth/logout` | 撤销当前会话 | `204` |

- 短信挑战绑定浏览器、手机号和签发时用户 ID，一次性消费并限制有效期、尝试次数和重发频率；未知手机号与错误验证码使用统一失败响应。
- 本地登录请求仅含 `username` 和 `password`。失败统一返回 `401 invalid_credentials`，不得透露账号是否存在、禁用或密码错误。
- 本地登录仅用于应急管理员，不提供注册、忘记密码或网页登录重置能力。
- 成功登录必须轮换会话标识。`GET auth/session` 不创建认证会话；无有效会话返回 `401 session_invalid`。
- `SessionResponse` 固定包含 `principal`、`content_models`、`csrf_token`、`idle_expires_at` 和 `expires_at`。`content_models` 只返回 Principal 具有至少一项有效模型权限且仍处于活动状态的模型摘要 `id`、`key` 和 `display_name`，不得借此放宽 `models.view`；`csrf_token` 只用于当前会话。
- 退出必须先撤销服务端会话，再清除 Cookie；重复提交已失效会话返回 `401 session_invalid`，不伪装为成功。

## 2. Cookie 和会话

- 会话 Cookie 名固定为 `cms_session`，值为不可解释的高熵会话标识，不得包含 Principal、权限或 Token 明文。
- Cookie 属性固定为 `HttpOnly; Secure; SameSite=Lax; Path=/api/admin/v1`，不得设置 `Domain`。
- 登录响应设置的 Cookie `Expires` 不得晚于会话绝对过期时间；退出使用同 Path、同安全属性和过去的 `Expires` 清除 Cookie。
- 空闲超时固定为 30 分钟，绝对有效期固定为自会话创建起 12 小时。访问有效的已认证管理 API 可以刷新服务端 `last_seen_at` 和 `idle_expires_at`，但不得延长 `expires_at`。
- 服务端判定 `now >= idle_expires_at` 或 `now >= expires_at` 时会话失效。过期、撤销、未知会话统一返回 `401 session_invalid`。
- 用户被禁用后，其现有会话立即视为无效，不等待缓存或下一次超时。重新启用用户不会恢复旧会话。
- 会话状态保存在 MySQL。Cookie 丢失、服务重启或切换实例不得改变上述语义。

## 3. CSRF 和 Origin

- 认证会话下所有非安全方法，即 `POST`、`PUT`、`PATCH`、`DELETE`，必须同时通过 Origin 和 CSRF 校验。
- CSRF 请求头固定为 `X-CSRF-Token`，值必须与当前服务端会话中的 Token 使用常量时间比较。不得从查询参数、请求体或 Cookie 替代读取。
- `GET auth/session` 返回 Token；前端只保存在进程内存，不写入 `localStorage`、`sessionStorage`、IndexedDB 或日志。
- Origin 比较以 `APP_BASE_URL` 的 origin 为唯一基准，严格比较小写 scheme、规范化 host 和有效端口。路径、查询和尾斜线不参与比较；默认端口按 scheme 规范化。
- 写请求缺少 `Origin` 返回 `403 origin_required`，不匹配返回 `403 origin_mismatch`。不使用 `Referer` 回退，也不通过宽松 CORS 白名单放行。
- 有效认证会话的写请求缺少或不匹配 CSRF Token，分别返回 `403 csrf_token_required` 或 `403 csrf_token_invalid`。
- 验证码挑战、短信登录和 `POST auth/local/login` 尚无认证会话，必须通过相同 Origin 检查，但不要求 CSRF Token。`POST auth/logout` 同时要求 Origin 和 CSRF。
- 校验顺序不得泄露会话存在性：先做 Origin，再认证会话，最后校验 CSRF。

## 4. Principal

`Principal` 是管理 API 唯一认证上下文 DTO：

```json
{
  "user_id": "usr_xxx",
  "display_name": "张三",
  "email": "zhangsan@example.com",
  "auth_method": "sms",
  "is_emergency_admin": false,
  "has_high_risk_role": true,
  "system_permissions": ["models.view", "models.create"],
  "model_permissions": [
    {
      "model_id": "mdl_xxx",
      "permissions": ["content.view", "content.create"]
    }
  ]
}
```

- `email` 可以为 `null`；不得用邮箱作为稳定身份键。
- `auth_method` 仅为 `sms` 或 `local`。
- `is_emergency_admin` 和 `has_high_risk_role` 是互相独立的服务端主体标记；业务能力不得从普通权限码反推。
- 两类权限数组均去重并按代码字典序返回；同一 `model_id` 只出现一次，模型授权按 `model_id` 排序。
- `Principal` 只包含当前请求的有效权限并集，不返回角色内部结构。服务端每次请求均重新确认会话和用户有效性。
- 前端可用权限改善导航和按钮展示，但服务端权限校验始终是安全边界。

## 5. 权限代码

系统权限代码固定为：

| 代码 | 含义 |
| --- | --- |
| `users.view` | 查看用户 |
| `users.manage` | 创建、修改、禁用用户 |
| `roles.view` | 查看角色及授权 |
| `roles.manage` | 创建、修改角色及绑定 |
| `models.view` | 查看模型和字段 |
| `models.create` | 创建模型和字段 |
| `models.update` | 修改模型和字段 |
| `models.archive` | 归档模型和字段 |
| `assets.view` | 查看素材 |
| `assets.upload` | 上传素材 |
| `assets.update` | 修改素材元数据 |
| `assets.archive` | 归档素材 |
| `api_keys.view` | 查看 API Key 元数据 |
| `api_keys.create` | 创建或轮换 API Key |
| `api_keys.revoke` | 撤销 API Key |
| `audit.view` | 查看审计事件 |

模型级权限代码固定为：`content.view`、`content.create`、`content.update`、`content.archive`、`content.submit`、`content.review`、`content.publish`、`content.unpublish`。

- 权限默认拒绝。除应急管理员和内置高危角色外，用户有效权限严格等于其全部有效普通角色授权的去重并集，不存在隐式用户权限或其他例外。
- 应急管理员是独立的 break-glass 身份：启用状态下固定拥有全部系统权限，以及全部活动模型的所有模型级权限。该授权不依赖角色，不能通过删除角色、解绑角色或清空角色授权撤销；禁用后不再拥有权限。
- 系统内置唯一、不可编辑和不可删除的高危角色 `high_risk_admin`。其持有者动态拥有全部系统权限及全部活动模型权限，并可向非应急用户授予或移除该角色；高危角色不能管理任何应急管理员账户。
- 系统模型管理权限和某一动态模型的内容权限不可互相替代。
- `content.review` 表示审核动作资格；审核授权只判断请求时有效的模型权限。
- 未知权限代码不得静默接受、持久化或返回。

## 6. 审计事件最小结构

领域服务追加审计事件时使用以下最小结构：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `id` | string | 不透明事件 ID |
| `occurred_at` | RFC 3339 UTC | 事件实际发生时间 |
| `request_id` | string | 与请求响应头相同；后台命令自行生成 |
| `actor_type` | `user`、`system` | 登录失败无法定位用户时使用 `system` |
| `actor_id` | string 或 null | `user` 必填，`system` 可空 |
| `actor_display_name` | string 或 null | `user` 保存事件发生时的名称快照且必填，`system` 必须为空 |
| `action` | string | 稳定 `snake_case` 事件代码 |
| `resource_type` | string | 稳定 `snake_case` 资源类型 |
| `resource_id` | string 或 null | 无已知资源时可空 |
| `result` | `success`、`failure` | 不使用模糊中间状态 |
| `ip` | string | 规范化客户端 IP |
| `user_agent` | string | 可截断，不得缺失为 null |
| `changes` | object | 非敏感变更摘要，无变更时 `{}` |
| `failure_code` | string 或 null | 成功必须为空；失败使用稳定错误码 |

- 成功事件与其业务状态在同一事务中追加，任一步失败整体回滚。
- 需要保留的失败尝试在业务事务回滚后独立追加，必须标记 `result=failure`，不得声称业务状态已改变。
- 审计存储只追加，不提供业务更新和删除接口。
- 用户以后改名不得改变历史事件中的 `actor_display_name`；查询直接返回快照，不动态关联当前用户名称。
- 密码、密码哈希、短信验证码、会话标识、CSRF Token、完整 API Key、对象存储凭证和签名 URL 不得进入任何审计字段。
- 认证动作代码为 `auth_sms_login_succeeded`、`auth_sms_login_failed`、`auth_local_login_succeeded`、`auth_local_login_failed`、`auth_logout_succeeded`；模型动作代码为 `model_created`、`model_updated`、`model_archived`、`model_field_created`、`model_field_updated`、`model_field_order_updated`、`model_field_archived`。

## 7. 模型 DTO 和字段类型

模型资源分为 `ContentModelSummary` 和带完整 `fields` 的 `ContentModel`。模型稳定字段为 `id`、`key`、`display_name`、`description`、`status`、`created_at`、`updated_at`；`status` 仅为 `active` 或 `archived`。

- `key` 使用小写字母开头，只含小写字母、数字和下划线，长度 1 至 64；创建后不可修改。
- `display_name` 长度 1 至 120；`description` 长度不超过 1000，缺省为空字符串。
- 模型归档使用 `DELETE /models/{model_id}`，只改变状态，不物理删除；归档模型不允许新增或修改字段。若仍有活动关联字段指向该模型，归档返回 `409 model_referenced`；归档和关联字段写入必须按模型 ID 稳定排序锁定相关模型并在事务内复核。
- 字段 `key` 在模型内唯一，遵循同一标识格式，创建后不可修改。
- 字段类型名称固定为：

```text
single_line_text
multi_line_text
rich_text
integer
decimal
boolean
date
datetime
single_select
multi_select
json
single_media
multi_media
single_relation
multi_relation
object
repeatable_group
```

字段公共属性为 `id`、`key`、`display_name`、`description`、`type`、`required`、`default_value`、`constraints`、`children`、`status`、`created_at` 和 `updated_at`。

关联字段的 `constraints.target_model_id` 必须指向另一个 active 模型，不允许关联字段所属模型自身；违反时返回 `409 target_model_self_relation`。

字段内容值及非空 `default_value` 的 JSON 表示固定如下；默认值仍须满足该字段 constraints：

| 字段类型 | JSON 表示 |
| --- | --- |
| `single_line_text`、`multi_line_text` | string |
| `rich_text` | 结构化 JSON object；F1 第一波不接受非空默认值 |
| `integer` | 无小数部分的 JSON integer |
| `decimal` | 不含指数、无多余前导零的十进制 string |
| `boolean` | boolean |
| `date` | `YYYY-MM-DD` string |
| `datetime` | RFC 3339 string，输出归一化为 UTC `Z` |
| `single_select` | 一个枚举 `value` string |
| `multi_select` | 枚举 `value` string 的无重复 array |
| `json` | 任意合法 JSON value |
| `single_media`、`multi_media` | 内容值在 F3 冻结；F1 第一波不接受非空默认值 |
| `single_relation`、`multi_relation` | 内容值在 F2 冻结；F1 第一波不接受非空默认值 |
| `object` | 以 child `key` 为属性名的 object |
| `repeatable_group` | 以 child `key` 为属性名的 object array |

- `description` 缺省为空字符串；`required`、`unique`、`filterable`、`sortable` 缺省均为 `false`；`default_value` 缺省为 `null`，表示不提供默认值。
- `default_value` 必须符合字段自身类型和全部 constraints；`required=true` 不禁止默认值为 `null`，但 `null` 表示无默认值而不是字段的实际默认内容。
- `min_length`、`max_length` 仅用于文本类型；`minimum`、`maximum` 仅用于 `integer` 和 `decimal`。
- `enum_options` 仅用于单选和多选，每项含稳定 `value` 和可修改 `label`；`value` 在字段内唯一，创建后不可修改。
- `target_model_id` 仅用于单关联和多关联，必须指向有效且未归档模型。
- `unique`、`filterable`、`sortable` 只允许根级标量字段。F1 根级标量为文本、整数、小数、布尔、日期、日期时间和单选；JSON、富文本、多选、媒体、关联、对象和重复组均不属于该集合。
- `children` 只允许用于 `object` 和 `repeatable_group`，其他类型必须为 `[]`。对象和重复组至少包含一个子字段。
- 根字段深度为 0；进入一个 `object` 或 `repeatable_group` 子级后深度加 1；允许最大深度为 2，深度 2 的字段不得再声明 children。
- 归档字段保留在模型详情中并标记 `status=archived`，默认值和历史内容不得被重写；归档字段不可恢复、修改或参与排序。活动字段使用独立的同级位置唯一约束并在响应中排在归档字段之前，归档位置不阻塞活动字段排序。
- 字段类型只有在该模型从未产生任何内容时才可修改。允许修改类型时，必须同时重新校验并显式提交所有受类型影响的属性。
- `decimal` 在 JSON 中使用字符串表示，格式为规范十进制，不接受指数形式，以避免浏览器和后端精度漂移。零固定为 `"0"`，负零归一化为 `"0"`，小数末尾零是否保留不具有业务含义。日期使用 full-date，日期时间遵循 F0 RFC 3339 规则。
- `rich_text` 的内容值是结构化 JSON 文档且禁止任意 HTML；具体文档节点契约随草稿内容 DTO 冻结。本波不得以字符串 HTML 作为默认值。
- 单媒体、多媒体和关联内容值的具体 ID 表达分别在 F3 和 F2 冻结；本波对应字段的 `default_value` 必须为 `null`。

## 8. 模型管理 API

模型路由固定为：

```text
GET, POST      /api/admin/v1/models
GET, PATCH,
DELETE         /api/admin/v1/models/{model_id}
GET, POST      /api/admin/v1/models/{model_id}/fields
PUT            /api/admin/v1/models/{model_id}/fields/order
POST           /api/admin/v1/models/{model_id}/fields/{parent_field_id}/children
GET, PATCH,
DELETE         /api/admin/v1/models/{model_id}/fields/{field_id}
```

- `POST /models` 要求 `models.create`；读取要求 `models.view`；修改要求 `models.update`；归档要求 `models.archive`。
- `PUT /models/{model_id}/fields/order` 要求 `models.update`；`parent_id=null` 表示根字段，否则表示直接父字段。`base_field_ids` 必须等于事务锁定后该父级全部活动字段的当前顺序，`field_ids` 必须是相同完整集合的新顺序；基线过期、遗漏、重复、跨父级、包含归档字段或无效父级均拒绝，状态冲突返回 `409 field_order_conflict`。排序与审计在同一事务原子提交。
- 非根字段排序的父字段必须是活动、深度小于 2 的 `object` 或 `repeatable_group`。字段排序成功追加 `model_field_order_updated` 审计事件。
- `POST /models/{model_id}/fields/{parent_field_id}/children` 要求 `models.update`，请求和响应分别为 `ContentFieldInput` 和 `ContentField`。服务端在同一事务内按全局一致顺序锁定当前模型和全部关联目标模型，再锁定完整字段树，重新校验模型、父字段状态、容器类型和深度，并将新字段追加到该父字段全部活动 children 的最大 position 之后；字段、关联校验和 `model_field_created` 审计原子提交。
- 根字段创建在模型和字段树加锁后追加到全部活动根字段末尾。
- 创建模型和创建字段分开执行，不提供假原子批量创建。对象或重复组的 children 随其父字段一次创建和更新。
- `PATCH` 使用显式字段的部分更新，不采用 JSON Merge Patch；请求中省略表示保持不变，显式 `null` 仅允许用于 schema 声明可空的字段。
- 模型和字段归档成功返回 `204`。已经归档的资源再次归档返回 `409 resource_archived`，不伪装为幂等成功。
- `GET /models` 默认返回 active 和 archived，可用 `status` 精确筛选；模型数量受内部规模约束，本波不冻结分页。
- 标准部署必须先停止旧应用进程，再执行 `cms migrate`，迁移成功后启动新版本。`serve` 的 pending migration gate 能阻止新进程在旧 schema 上启动，但不能约束仍在运行的旧进程；`000036`/`000037` 字段顺序迁移不得与旧版本写入并发执行。

## 9. 校验详情

`validation_failed` 的 `error.details` 元素固定为：

```json
{
  "path": "/constraints/max_length",
  "code": "out_of_range",
  "message": "max_length 必须大于或等于 min_length"
}
```

- `path` 是相对于请求 JSON 根的 RFC 6901 JSON Pointer；请求体整体错误使用空字符串，数组位置使用数字段，例如 `/children/0/key`。
- `code` 是稳定的非空 `snake_case` 机器码；前端只依赖 `path` 和 `code`，不依赖 `message`。
- `message` 是面向用户的非空说明，可调整文案。
- 同一请求返回所有可安全确定的独立校验错误，按 `path`、`code` 字典序稳定排序。
- 语法错误、未知属性、格式、类型、范围、组合约束和嵌套深度错误均可使用该结构；权限、认证、未找到和冲突错误的 details 保持空数组。

## 10. 前端 API 消费规则

- 前端只通过统一 API Client 消费 OpenAPI DTO，base URL 固定使用同源相对路径 `/api/admin/v1`，所有请求设置 `credentials: "same-origin"`。
- 登录成功或应用启动后调用 `GET /auth/session`，将 `Principal` 和 CSRF Token 仅保存在内存。页面刷新后重新获取，不自行解析 Cookie。
- Client 对所有非安全方法自动附加当前 `X-CSRF-Token`。收到 `401 session_invalid` 时清空内存认证状态并转到登录页；不得对写请求自动重放。
- 仅按 `error.code` 分支处理错误；`validation_failed` 按 JSON Pointer 映射字段，无法映射的详情显示为表单级错误；始终展示或保留 `request_id` 供诊断。
- `403` 表示已认证但不允许，不能一律当作会话过期。前端权限隐藏不代替服务端授权。
- 不使用浏览器存储保存密码、会话值、CSRF Token、OIDC Token 或完整 Principal 权限快照。
- DTO 中的 ID 始终作为不透明字符串，时间按 UTC RFC 3339 解析，`decimal` 保持字符串直到交给明确的十进制实现。
- Client 不推断 OpenAPI 未声明的响应字段，不把未知状态码当成功，不调用尚未聚合且运行时未实现的分片路径。

## 11. ContentExistenceChecker

模型模块依赖内容模块提供以下语义接口；名称和方法签名作为跨模块冻结点：

```go
type ContentExistenceChecker interface {
	HasAnyContent(ctx context.Context, q database.Querier, modelID string) (bool, error)
}
```

- `true` 表示该模型曾创建至少一个内容条目；已归档条目仍计入。是否存在当前草稿、已发布版本或投影不影响结果。
- `false` 只表示该模型从未创建内容条目。历史 Revision 不应脱离条目单独存在；实现不得用“当前可见内容为空”替代该判断。
- 模型 Service 在字段类型改变、字段约束收紧或启用需回填的查询能力前调用该接口；返回 `true` 时分别拒绝并返回 `409 field_type_locked`、`409 field_schema_migration_required` 或 `409 field_projection_backfill_required`，返回 error 时失败关闭且不得写入任何模型变更。
- 调用方先验证模型存在且属于当前操作，checker 不负责认证、授权、模型查找或字段校验；`q` 必须是模型写事务的当前执行器，不得回退全局连接池。
- 实现不得使用可能过期的进程内缓存。内容创建与字段类型变更必须通过数据库锁或等价事务约束串行化，避免检查后并发创建内容的竞态。

## 12. 第二波用户契约

用户路由固定为：

```text
GET, POST        /api/admin/v1/users
GET, PATCH       /api/admin/v1/users/{user_id}
PATCH            /api/admin/v1/users/{user_id}/phone
PUT              /api/admin/v1/users/{user_id}/roles
```

- 管理 API 可创建短信用户，但不提供用户删除、应急管理员创建、身份解绑或本地密码管理 API。应急管理员资格和密码只能通过 `cms admin ensure/reset-password` 管理。
- 用户响应字段固定为 `id`、`display_name`、`email`、`phone_masked`、`phone`、`auth_methods`、`is_emergency_admin`、`has_high_risk_role`、`status`、`role_ids`、`created_at` 和 `updated_at`。列表项不含 `phone` 和 `role_ids`；`phone` 在管理 API 响应中固定为 `null`；`auth_methods` 去重并按字典序返回。历史仅 OIDC 用户保留为不可登录主体并返回空 `auth_methods`，不得伪造 local 或 sms 凭证。
- `status` 仅为 `enabled` 或 `disabled`。`PATCH` 只接受 `status`，不承担资料同步；禁用立即使全部现有会话失效，重新启用不恢复旧会话。
- 查看要求 `users.view`；启禁用要求 `users.manage`。读取用户详情中的角色还要求 `roles.view`，因此 `GET` 用户详情和返回完整 `User` 的 `PATCH` 启禁用都必须同时满足各自操作权限与 `roles.view`；否则不得通过裁剪 `role_ids` 部分返回，整个请求返回 `403`。
- 系统必须始终至少保留一个 `status=enabled`、具有有效本地凭据且可通过应急入口登录的应急管理员。禁用最后一个此类用户返回 `409 last_emergency_admin_required`。配置关闭网页登录不改变账号的应急管理员身份，也不解除该约束。
- 普通主体和高危角色都不能禁用、改凭证、改角色或改变应急管理员资格；应急管理员可以管理其他应急管理员，但不能禁用最后一个有效应急管理员。`admin ensure` 对已存在且启用的应急管理员不得修改密码或撤销会话；`admin reset-password` 必须更新密码、重新启用目标并撤销其全部现有会话。本地会话创建前必须在同一事务重新核对用户启用状态、应急资格和已验证密码哈希，防止旧密码在重置提交后创建新会话。
- `PUT /users/{user_id}/roles` 原子替换用户的完整角色集合，要求 `roles.manage`；空数组表示移除全部角色。未知、已删除或重复角色 ID 返回 `400 validation_failed`，不得部分绑定。
- 用户列表可按 `status`、`auth_method` 和不区分大小写的 `query` 筛选；`query` 匹配显示名称、邮箱或脱敏手机号，不匹配历史 OIDC subject。列表按 `updated_at DESC, id DESC` 稳定排序并采用第 16 节游标语义。

## 13. 第二波角色和授权契约

角色路由固定为：

```text
GET, POST        /api/admin/v1/roles
GET, PATCH,
DELETE           /api/admin/v1/roles/{role_id}
PUT              /api/admin/v1/roles/{role_id}/system-permissions
PUT              /api/admin/v1/roles/{role_id}/model-permissions
```

- 角色字段固定为 `id`、`key`、`kind`、`display_name`、`description`、`system_permissions`、`model_permissions`、`created_at` 和 `updated_at`。`kind` 为 `custom` 或 `high_risk`；`key` 遵循模型 key 格式、全局唯一且创建后不可修改；显示名称长度 1 至 120，说明最长 1000。
- 角色读取要求 `roles.view`，创建、修改、删除、绑定和授权要求 `roles.manage`。除第 5 节定义的应急管理员和内置高危角色外，权限默认拒绝，用户有效权限严格为其全部普通角色授权的去重并集。
- 创建角色不隐式授予权限。两条授权 `PUT` 均原子替换该角色对应类型的完整授权集合，空数组表示清空；未知权限代码、重复项、未知或归档模型返回 `400 validation_failed`，不得部分写入。
- 普通角色管理员不能修改自己的角色绑定或当前分配给自己的角色，也不能管理当前权限高于自己的目标用户或角色；创建用户、替换用户角色和替换角色授权时，现有及目标权限都必须是操作者有效权限的子集。应急管理员和高危角色不受普通权限子集限制。
- 用户与角色写操作必须在同一事务内锁定并重新加载操作者当前主体和角色授权后再鉴权，不能只信任 HTTP 认证阶段捕获的 Principal；角色父行及其系统、模型授权行必须使用锁定读，避免撤权后的慢请求或 REPEATABLE READ 旧快照继续写入。
- 用户与角色安全事务仅对 MySQL 1213 死锁受害事务自动重试，最多执行 3 次完整事务；包含对象存储等外部副作用的通用事务以及其他数据库错误和业务错误不得重试。
- 已激活短信登录手机号只能由高危角色或应急管理员替换，且目标不得是应急管理员；该能力由事务内主体等级判断，不能从 `users.manage` 等普通权限码反推。换号成功必须在同一事务撤销目标全部会话，审计仅记录变更前后的脱敏手机号。
- SMS 会话创建必须在事务内锁定用户并重新核对挑战手机号仍是当前登录凭证，避免换号已撤销会话后旧手机号挑战再创建新会话。管理 API 的通用用户响应固定返回 `phone: null`，所有主体仅可见脱敏手机号，完整登录凭证不得通过管理 API 回传。
- 模型授权按 `model_id` 唯一，每项权限去重；响应中的系统权限按代码排序，模型授权按 `model_id` 排序，其中权限按代码排序。
- 删除角色成功返回 `204`，并在同一事务移除用户绑定和该角色授权；角色不存在返回 `404`。角色不采用归档状态，删除后其 ID 不得复用。
- 任何角色或授权变更不得绕过第 12 节的应急管理员约束。应急管理员资格来自有效本地凭据和启用状态，不以某个角色的名称或 ID 判定。高危角色持有者不能禁用、改凭证、改角色或以其他方式管理应急管理员。

## 14. 第二波审计查询契约

审计只读路由固定为：

```text
GET              /api/admin/v1/audit/events
GET              /api/admin/v1/audit/events/{event_id}
```

- 两条路由均要求 `audit.view`。不提供创建、修改、删除、批量清理或重放 API。
- 列表可按 `actor_type`、`actor_id`、`action`、`resource_type`、`resource_id`、`result`、`occurred_from` 和 `occurred_to` 精确筛选；时间区间两端均包含，且 `occurred_from` 不得晚于 `occurred_to`。
- 列表按 `occurred_at DESC, id DESC` 稳定排序并采用第 16 节游标语义。响应仍使用第 6 节完整 `AuditEvent`，不得因列表查询放宽敏感字段过滤。
- 未知事件返回 `404`。查询本身不得产生审计事件，避免只读访问递归扩增审计表；访问日志仍可记录非敏感请求元数据。

## 15. 第二波草稿内容契约

草稿内容路由固定为：

```text
GET, POST        /api/admin/v1/models/{model_id}/entries
GET, PATCH,
DELETE           /api/admin/v1/models/{model_id}/entries/{entry_id}
GET              /api/admin/v1/models/{model_id}/entries/{entry_id}/revisions
GET              /api/admin/v1/models/{model_id}/entries/{entry_id}/revisions/{revision_id}
```

- 条目字段固定为 `id`、`model_id`、`status`、`current_draft_revision_id`、`created_by`、`created_at`、`updated_at` 和详情中的 `current_draft_revision`。`status` 仅为 `draft` 或 `archived`。
- Revision 字段固定为 `id`、`entry_id`、`model_id`、`number`、`content`、`created_by` 和 `created_at`。`number` 从 1 开始在条目内严格递增；Revision 不可修改或删除。
- `content` 是以模型根字段 `key` 为属性名的 JSON object。服务端必须根据当前模型字段树校验未知字段、必填、类型、constraints、枚举、嵌套深度和归档字段；失败返回 `400 validation_failed` 且不得产生 Revision。
- 富文本值遵循 OpenAPI `RichTextDocument`：根节点固定为 `doc`；块节点仅允许 `paragraph`、`heading`、`bullet_list`、`ordered_list`、`list_item`、`blockquote`、`code_block`；行内节点仅允许 `text`、`hard_break`；mark 仅允许 `bold`、`italic`、`underline`、`strike`、`code`。所有节点递归拒绝白名单外属性，不提供 HTML、脚本、事件处理器、链接、URL 或任意 attrs。第一阶段尚未实现媒体与关联持久化：`single_media`、`multi_media`、`single_relation`、`multi_relation` 字段只接受缺省或 JSON `null`；任何非 `null` 值均返回 `400 validation_failed`，详情代码固定为 `unsupported_non_null_value`。若该字段 `required=true`，缺省或 `null` 仍按必填失败，因此此类字段在能力实现前不能保存有效草稿。
- 创建条目必须在同一事务创建第一个不可变 Revision 并设置 `current_draft_revision_id`；成功返回 `201`。编辑只接受完整 `content` 替换，每次有效保存都在同一事务创建新 Revision 并切换指针；不提供原地 JSON Patch，也不把内容写入条目行。
- `current_draft_revision_id` 是 API DTO 名称；数据库使用独立 `content_draft_pointers` 保存指针，并以 `(revision_id,entry_id,model_id)` 复合外键保证 Revision 归属。写事务统一先锁 model、再锁 entry，首个 Revision 在 entry 创建后插入，最后建立指针，不形成循环插入依赖。
- 根级 `unique=true` 标量的非 `null` 值在同一事务原子替换 `content_unique_values` 占用。先由字段类型标签、NUL 分隔符和规范值组成类型安全 canonical 输入：文本及单选保留原字符串，整数使用十进制整数，小数使用最简有理数，布尔使用 `true`/`false`，日期保留 full-date，日期时间使用归一化 UTC；`canonical_value` 固定保存该输入的 SHA-256。唯一约束固定为 `(model_id,field_id,canonical_value)`，并发冲突由 MySQL 唯一约束裁决，返回 `400 validation_failed`，字段详情代码为 `not_unique`。
- 编辑使用必填 `base_revision_id` 做乐观并发检查，必须等于当前草稿指针；不匹配返回 `409 draft_revision_conflict`，不得创建 Revision。即使新内容与当前 JSON 相同，一次通过校验的显式保存仍产生新 Revision。
- 归档条目返回 `204`，改变条目状态并释放全部唯一值占用，不创建 Revision且不物理删除历史；释放后其他草稿可使用原值。重复归档返回 `409 resource_archived`。归档条目及其 Revision 仍可读取，但不可编辑或恢复。
- 列表默认只返回 `draft`，可按 `status` 精确筛选；按 `updated_at DESC, id DESC` 稳定排序并采用第 16 节游标语义。Revision 列表按 `number DESC` 排序，也采用相同分页外壳。
- 条目读取和 Revision 读取要求目标模型的 `content.view`；创建要求 `content.create`；编辑要求 `content.update`；归档要求 `content.archive`。系统级模型权限不可替代这些模型级内容权限。
- 创建、编辑和归档分别追加 `content_entry_created`、`content_revision_created` 和 `content_entry_archived` 成功审计事件；业务状态、Revision、当前指针和成功审计处于同一事务。
- 本阶段不提供提交、审核、驳回、发布、下线、恢复、Revision 回滚、关联展开、内容过滤或总数统计。未列出的路由保持 `404`。

## 16. 游标分页最小语义

- 用户、审计事件、草稿条目和 Revision 列表使用统一外壳 `{items, next_cursor}`；`next_cursor` 为不透明 string 或 `null`。
- `limit` 默认 20，最小 1，最大 100。返回项少于 `limit` 时 `next_cursor` 必须为 `null`；否则它表示当前结果最后一项之后的位置。
- 游标至少绑定资源类型、排序键和全部筛选条件。调用方修改筛选条件或把游标用于其他列表时返回 `400 invalid_cursor`；游标格式、签名和内部字段不属于公开契约。
- 分页不提供跨请求快照隔离。并发创建或更新可能使后续页反映新状态，但服务端必须使用声明的唯一稳定排序键继续查询，不得在数据未变化时重复或跳过项目。

## 17. 第二波稳定错误和决策

- 第二波新增稳定错误码为 `last_emergency_admin_required`、`draft_revision_conflict`、`invalid_cursor` 和 `resource_archived`；动态字段的不支持值使用校验详情代码 `unsupported_non_null_value`。
- 基于恢复目标，应急管理员是独立 break-glass 身份并固定拥有全部系统权限及全部活动模型权限；角色系统不能撤销该能力。内置高危角色是可审计传播的日常最高权限能力，但不能反向管理应急管理员；其他用户严格采用普通角色授权并集。
- 短信用户由管理 API 创建；历史 OIDC 身份只保留数据关联，不再用于登录。角色绑定和授权采用完整集合原子替换，避免增删接口并发后留下调用方未知状态。
- 内容编辑采用完整 JSON 加草稿基线指针，不在动态 JSON 上冻结 Merge Patch 语义。Revision 是保存历史，条目是生命周期和当前指针，两者职责不合并。
- 第二波用户、角色、审计和草稿内容 OpenAPI 分片已经完成实现并聚合进根 OpenAPI。
