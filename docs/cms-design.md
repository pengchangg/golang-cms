# 企业内部 CMS 设计文档

## 1. 文档状态

- 状态：V1 已实施，本文描述当前运行架构
- 目标版本：V1
- 开发计划：[多 Agent 开发计划](./development-plan.md)
- 部署形态：单个应用 Docker 镜像，外接 MySQL 8.0 与私有 S3 兼容对象存储
- 目标规模：少于 100 名后台用户、百万级内容、内容 API 峰值低于 200 QPS

## 2. 建设目标

建设一个企业内部使用的动态内容管理系统。系统包含 Go 后端和 React 管理网页，业务人员可以通过可视化界面创建内容模型、维护内容、提交审核和发布；内部客户端通过标准 REST API 和 API Key 读取已发布内容。

V1 必须具备：

- 大陆手机号短信登录和本地应急管理员登录。
- 用户、角色和模型级操作权限。
- 可视化动态内容模型。
- 内容增删改查、版本管理和固定审核流程。
- 面向客户端的只读 REST API。
- API Key 签发、限制、过期、撤销和轮换。
- 私有 S3 兼容对象存储素材库。
- CSV 导入导出。
- 不可变审计日志。
- OpenAPI 3.1 接口文档。
- 数据库迁移、初始化命令和应用 Dockerfile。

## 3. 非目标

V1 不包含：

- 多租户。
- 部门或组织树。
- 字段级权限和数据范围权限。
- 自定义审批流程编排。
- 多语言内容。
- 页面装修器或低代码页面生成器。
- 插件系统。
- 全文搜索引擎。
- GraphQL。
- 微服务拆分。
- Redis、Kafka、RabbitMQ 等额外基础设施。
- SQLite 或单容器内置数据库。
- 任意深度的字段嵌套。
- 为所有动态字段自动建立数据库索引。

## 4. 总体架构

系统采用模块化单体。Go 服务同时提供管理 API、客户端内容 API、认证端点和 React 静态资源。

```text
+-----------------+                       +--------------------+
| React 管理网页  +---------------------->| Go CMS 单体        |
+-----------------+                       |                    |
                                          | /api/admin/v1      |
                     +-----------------+   | /api/content/v1    |
                     | 腾讯云短信服务  |<--+ /openapi           |
                     +-----------------+   +------+--------+-----+
+-----------------+                              |        |
| 内部业务客户端  +------- API Key --------------+
+-----------------+
                                                 |        |
                                        +--------v---+ +--v----------+
                                        | MySQL 8.0  | | S3 兼容存储 |
                                        +------------+ +-------------+
```

管理网页与 API 同源部署，以降低 CORS、Cookie 和运维配置复杂度。Go 应用保持无状态，业务状态和会话保存在 MySQL 中，以支持多实例横向扩容。

## 5. 技术选型

| 层级 | 选型 |
| --- | --- |
| 后端 | Go，基于标准 `net/http` 兼容路由 |
| 数据库 | MySQL 8.0、事务、JSON、数据库迁移 |
| 数据访问 | 显式 SQL 查询层 |
| 管理端 | React、TypeScript、Vite、Ant Design |
| API 描述 | OpenAPI 3.1 |
| 管理端认证 | 手机号短信验证码、本地应急登录、服务端会话 Cookie |
| 应急认证 | 本地管理员、Argon2id 密码哈希、命令行重置 |
| 客户端认证 | API Key，数据库仅保存密钥哈希 |
| 素材存储 | 私有 S3 兼容对象存储 Bucket、签名上传和签名下载 |
| 导入导出 | HTTP 请求内同步流式 CSV |
| 部署 | 多阶段 Dockerfile，最终镜像包含后端与前端静态资源 |
| 日志 | JSON 结构化日志、请求 ID、操作者与耗时 |
| 时间规范 | 数据库存 UTC，API 使用 RFC 3339 |

具体依赖版本在项目初始化时固定，并提交 Go 与前端锁文件，不使用浮动版本。

## 6. 后端模块

后端按业务边界拆包，但编译为一个二进制：

| 模块 | 职责 |
| --- | --- |
| `auth` | 短信登录、本地应急登录、会话与退出 |
| `identity` | 用户、角色与角色绑定 |
| `permission` | 系统权限与模型操作权限校验 |
| `schema` | 动态模型、字段定义、校验规则与索引声明 |
| `content` | 内容、版本、关联、查询与工作流 |
| `asset` | 对象存储上传、确认、下载与素材引用 |
| `client` | API Key、模型访问范围、撤销与轮换 |
| `configuration` | 非秘密业务配置、命名空间级授权、版本工作流与客户端读取 |
| `transfer` | 同步 CSV 导入导出与错误报告 |
| `audit` | 不可变审计事件 |
| `platform` | 配置、数据库、迁移、HTTP 与日志 |

模块间通过公开服务接口调用。HTTP 处理器不得绕过所属模块直接修改其他模块的数据表。

## 7. 身份认证

### 7.1 手机号短信登录

- 普通管理账户由管理员预先创建并绑定中国大陆手机号，不开放自助注册。
- 登录先完成 CAPTCHA，再发送并验证一次性短信验证码。
- 生产短信由腾讯云发送；固定验证码只允许回环地址上的 development 环境。
- 管理端使用 `HttpOnly`、`SameSite=Lax` 会话 Cookie；外部访问 URL 为 HTTPS 时同时设置 `Secure`。
- 写操作执行同源检查和 CSRF Token 校验。
- 会话同时设置空闲超时和绝对过期时间。
- 用户被禁用后，现有会话立即失效。

### 7.2 本地应急管理员

- 不开放注册、忘记密码或邮件重置。
- 使用应用命令创建或重置，例如 `cms admin reset-password`。
- 密码使用 Argon2id 哈希。
- 应急登录入口与短信登录入口分离。
- 登录成功、登录失败和密码重置均写入审计日志。
- 系统必须至少保留一个有效应急管理员。
- 可以通过配置关闭应急账号的网页登录，但保留容器内命令行恢复能力。

## 8. 权限模型

系统采用 RBAC。用户可绑定多个角色，有效权限为所有角色权限的并集，未明确授权的操作默认拒绝。

### 8.1 系统权限

| 资源 | 操作 |
| --- | --- |
| 用户 | 查看、管理 |
| 角色 | 查看、管理 |
| 模型 | 查看、创建、修改、归档 |
| 素材 | 查看、上传、修改、归档 |
| API Key | 查看、创建、撤销 |
| 审计 | 查看 |
| 导入导出 | 执行、下载 |

### 8.2 内容模型权限

每个动态内容模型独立授权以下操作：

- 查看。
- 新增。
- 编辑。
- 删除或归档。
- 提交审核。
- 审核。
- 发布。
- 下线。

服务端必须在每个请求中校验权限。管理端隐藏按钮只用于改善体验，不能作为安全边界。

## 9. 动态内容模型

### 9.1 字段类型

V1 支持：

- 单行文本。
- 多行文本。
- 富文本。
- 整数。
- 小数。
- 布尔值。
- 日期。
- 日期时间。
- 单选枚举。
- 多选枚举。
- JSON。
- 单媒体。
- 多媒体。
- 单关联。
- 多关联。
- 对象。
- 可重复字段组。

### 9.2 字段属性

- 稳定字段标识。
- 显示名称。
- 帮助说明。
- 是否必填。
- 默认值。
- 字符串长度限制。
- 数字最小值和最大值。
- 枚举选项。
- 是否唯一。
- 是否可筛选。
- 是否可排序。

### 9.3 模型约束

- 模型标识和字段标识创建后不可直接修改，只能修改显示名称。
- 已存在内容后不能直接改变字段的数据类型。
- 删除模型或字段采用归档，不立即物理删除。
- 对象和可重复字段组最多嵌套两层。
- 唯一、筛选和排序仅支持根级标量字段。
- 富文本保存结构化 JSON，不接受未经处理的任意 HTML。
- 关联必须指向未归档模型中的有效内容。
- 已被内容或素材引用的对象不能直接永久删除。

## 10. 动态内容存储

不为每个动态模型创建独立 MySQL 业务表，也不依赖频繁动态 DDL。

采用 JSON 与类型化投影混合存储：

- 完整内容保存在版本表的 MySQL `JSON` 列中。
- 声明为唯一、可筛选或可排序的根级标量字段同步写入类型化投影表。
- 关联字段写入独立关联表。
- 媒体字段写入独立素材引用表。
- JSON、投影、关联和媒体引用在同一个事务中写入。
- 投影表通过模型、字段和值建立组合索引和唯一约束。

投影表按类型提供字符串、数字、布尔、日期时间等列，避免常用查询依赖 `JSON_EXTRACT` 全表扫描。

内容查询规则：

- 默认每页 20 条，单次最多 100 条。
- 仅允许使用模型中声明为可筛选的字段过滤。
- 仅允许使用声明为可排序的字段排序。
- 关联展开默认关闭，显式请求时最多展开一层。
- 限制单次查询的过滤字段数量和关联条件数量。
- 内容列表使用游标分页。
- 管理端可以额外返回受限的总数统计。

## 11. 内容版本和工作流

V1 使用固定审核流程：

```text
草稿 --提交--> 待审核 --通过--> 已发布
  ^                |
  |                +--驳回--> 已驳回
  |                              |
  +----------继续编辑------------+

已发布 --编辑--> 新草稿
已发布 --下线--> 已下线
已下线 --编辑--> 草稿
```

规则：

- 每次保存产生一个不可变内容版本。
- 内容记录保存当前草稿版本和当前已发布版本的指针。
- 编辑已发布内容不会立即覆盖线上版本。
- 审核通过时原子切换当前已发布版本。
- 驳回必须填写理由。
- 审核动作只按请求时有效的模型权限授权，提交人具备权限时可以审核自己提交的版本。
- 发布、驳回和下线均写入审计日志。
- 客户端内容 API 只读取当前已发布版本。
- 关联内容未发布时，客户端 API 不展开该关联。
- 被已发布内容引用的素材不能删除，只能归档。

## 12. REST API

管理 API 与客户端内容 API 分离，并分别生成 OpenAPI 文档。

### 12.1 管理 API

管理 API 使用后台会话认证：

```text
/api/admin/v1/auth/*
/api/admin/v1/users
/api/admin/v1/roles
/api/admin/v1/models
/api/admin/v1/models/{model}/fields
/api/admin/v1/models/{model}/entries
/api/admin/v1/models/{model}/entries/{id}
/api/admin/v1/models/{model}/entries/{id}/submit
/api/admin/v1/models/{model}/entries/{id}/approve
/api/admin/v1/models/{model}/entries/{id}/reject
/api/admin/v1/models/{model}/entries/{id}/unpublish
/api/admin/v1/assets
/api/admin/v1/imports
/api/admin/v1/exports
/api/admin/v1/api-keys
/api/admin/v1/configurations
/api/admin/v1/configurations/{namespace}/items
/api/admin/v1/audit-events
```

### 12.2 客户端内容 API

内容 API 使用 API Key，仅提供已发布内容的只读访问：

```text
GET /api/content/v1/models
GET /api/content/v1/models/{model}
GET /api/content/v1/models/{model}/entries
GET /api/content/v1/models/{model}/entries/{id}
GET /api/content/v1/assets/{id}
GET /api/content/v1/configurations/{namespace_key}
GET /api/content/v1/configurations/{namespace_key}/{item_key}
```

规则：

- 使用 `Authorization: Bearer <api-key>`。
- API Key 可以分别限制允许读取的内容模型和配置命名空间。
- 完整密钥只在创建时显示一次。
- 数据库只保存带独立盐值的密钥哈希和用于定位的短前缀。
- 支持名称、过期时间、最后使用时间、撤销和轮换。
- 只返回当前已发布版本。
- 内容 API 不提供任何写操作。
- 支持 `ETag` 和条件请求。
- API 版本通过 URL 路径管理。
- 返回统一错误结构、请求 ID 和稳定错误码。

配置命名空间保存可交付给内部客户端的非秘密业务配置，不用于密码、Token、私钥或基础设施凭据。配置项沿用草稿、审核、发布和下线工作流；客户端既可按 `namespace_key` 一次读取命名空间内全部当前发布项，也可按 `namespace_key/item_key` 读取单项。配置命名空间 scope 与模型 scope 分开校验；创建 API Key 当前仍要求至少授权一个模型，配置中的内容引用还要求目标模型位于同一 API Key 的模型 scope。两个读取方式均支持 `ETag` 和 `304`。

统一错误结构：

```json
{
  "error": {
    "code": "validation_failed",
    "message": "请求数据校验失败",
    "request_id": "req_xxx",
    "details": []
  }
}
```

## 13. 素材库

### 13.1 上传流程

1. 管理端向后端申请创建上传任务。
2. 后端校验权限、文件名、MIME 类型和文件大小。
3. 后端生成 S3 兼容对象存储限时上传签名。
4. 浏览器直接上传到私有 Bucket。
5. 管理端调用确认接口。
6. 后端通过标准 S3 用户元数据确认对象存在并保存素材记录；SHA-256 使用 `Metadata["sha256"]`，HTTP 表示为 `x-amz-meta-sha256`。
7. 内容字段只保存素材 ID，不保存临时签名 URL。

### 13.2 下载流程

- 管理端或客户端访问 CMS 素材地址。
- CMS 校验管理会话或 API Key。
- CMS 返回短时有效的对象存储签名地址或 HTTP 重定向。

### 13.3 安全规则

- 对象存储 Bucket 必须为私有。
- 对象 Key 由服务端生成，不直接使用用户文件名。
- 限制文件大小和允许的 MIME 类型。
- 文件扩展名不能代替 MIME 校验。
- 记录 SHA-256、大小、类型、上传者和不透明 ETag，不将 ETag 解释为内容摘要。
- 素材具有隔离、可用和归档状态，为后续病毒扫描预留接口。
- 病毒扫描不属于 V1 范围。

## 14. CSV 导入导出

### 14.1 导入

- 每个内容模型可以下载 CSV 模板。
- 第一行使用稳定字段标识，不使用显示名称。
- 上传 CSV 后在当前管理请求内流式解析、校验和写入。
- 默认整批校验通过后导入，避免半成功状态。
- 错误报告包含行号、字段、错误码和原因。
- 导入内容默认为草稿，不能绕过审核直接发布。
- 关联字段使用目标内容 ID。
- 媒体字段使用素材 ID。
- 对象和可重复字段组使用 JSON 字符串表示。

### 14.2 导出

- 按当前模型和筛选条件在当前管理请求内同步生成。
- 生成带 UTF-8 BOM 的 CSV，便于 Excel 打开。
- 导出最多 10,000 行、50 MiB，生成和传输共享 120 秒期限。
- 全进程最多 2 个且同一用户最多 1 个并发导出，临时文件在请求结束后删除。
- 导入和导出均在执行前校验用户权限并记录审计日志。

V1 不创建 CSV 任务表，不启动 Worker，也不使用对象存储保存导出文件。

## 15. 审计日志

必须审计以下事件：

- 短信和本地账号登录成功、登录失败与退出。
- 用户、角色和权限变更。
- 模型和字段变更。
- 内容新增、修改、提交、审核、驳回、发布、下线与归档。
- 素材上传、下载与归档。
- API Key 创建、撤销和异常使用。
- CSV 导入、导出与下载。
- 应急管理员密码重置。

审计事件至少包含：

- 事件 ID。
- UTC 时间。
- 请求 ID。
- 操作者类型和 ID。
- 动作。
- 资源类型和 ID。
- 执行结果。
- IP。
- User-Agent。
- 变更摘要。
- 失败原因。

审计表只允许追加，不向业务应用暴露更新和删除操作。密码、短信验证码、会话、对象存储凭证、完整 API Key 和其他敏感字段不得写入审计详情。

关键唯一约束：

- 投影字段按模型、字段和值建立索引。
- 动态唯一字段由投影表唯一约束保证。
- API Key 前缀用于定位，密钥哈希用于验证。

- 短信登录与本地应急登录。

- 模型字段设计器。
- 动态内容列表。

管理端以桌面端为主，同时保证平板和手机可以完成登录、查看和审核等关键操作。复杂模型设计在窄屏下使用简化编辑或只读提示，不强行压缩桌面布局。

生产启动前强制检查：

- 会话密钥长度符合要求。
- 应用基础地址使用 HTTPS。
- 短信提供方配置完整；固定验证码只用于受限开发环境。
- MySQL 可连接且版本为 8.0。
- S3 兼容对象存储 Endpoint、Region 和 Bucket 匹配，Bucket 可访问且不存在公开读写权限，凭证具备所需对象操作权限。
- 数据库迁移状态有效。
- 至少存在一个应急管理员。

对象存储 Adapter 使用 AWS SDK for Go v2 的 S3 客户端和 PresignClient，不得手写请求签名。阿里云部署使用官方 S3 兼容 Endpoint 和 virtual-host style，即 `S3_USE_PATH_STYLE=false`。凭证不得写入镜像、仓库、日志或前端构建产物。

## 19. 性能与容量约束

- 少于 100 名后台用户。
- 支持百万级内容记录。
- 内容 API 峰值低于 200 QPS。
- 常用索引查询服务端 P95 目标低于 300ms，不含外部网络和对象存储下载。
- 单个 CSV 默认限制 100,000 行，超过限制后拆分处理。
- 内容 API 单页最多 100 条。
- 通过无状态 Go 实例支持横向扩容。

正式交付前使用代表性的百万级数据验证投影查询。如果复杂多条件关联查询无法达到目标，再基于压测结果增加缓存，不在 V1 中提前引入 Redis。

## 20. 交付阶段

### 阶段一：基础平台与管理闭环

- 初始化 Go、React、MySQL 迁移和 Dockerfile。
- 完成短信登录、本地应急账号、会话、用户、角色与权限。
- 完成动态模型、动态表单和草稿内容 CRUD。
- 完成基础审计日志和 OpenAPI。

完成定义：系统可以独立部署，管理员可以登录、授权、创建模型并录入草稿。

### 阶段二：审核发布与客户端 API

- 完成不可变版本、固定审核流程和审核队列。
- 完成只读内容 API、API Key、过滤、排序、分页和 ETag。
- 完成字段投影、唯一约束和关联字段。

完成定义：内容经过审核发布后，内部客户端能够通过受限 API Key 稳定读取。

### 阶段三：素材与数据交换

- 完成私有 S3 兼容对象存储素材库和媒体字段。
- 完成 CSV 模板、导入、错误报告、导出和下载。
- 补齐素材与同步 CSV 导入导出的审计事件。

完成定义：业务人员可以管理素材并使用 CSV 批量交换内容，V1 全部能力可用。

三个阶段均可独立合并和部署，不依赖下一阶段才能正常工作。

## 22. 风险与应对

| 风险 | 应对 |
| --- | --- |
| MySQL 动态 JSON 查询性能不足 | 使用类型化投影表，避免高频 `JSON_EXTRACT` 全表扫描 |
| 模型频繁变更导致数据不一致 | 锁定标识和已有字段类型，删除采用归档 |
| 对象存储暂时不可用 | 元数据保持可访问，上传下载返回可重试错误，不破坏数据库状态 |
| 同步 CSV 请求耗尽资源 | 行数、字节数、期限、进程级和用户级并发限制 |
| API Key 泄露 | 仅保存哈希，限制模型范围，支持过期、撤销和轮换 |
| 大 CSV 占用过多内存 | 流式解析和分批校验，不将整个文件载入内存 |
| 应用回滚遇到数据库差异 | 使用向前兼容的扩展式迁移，不自动回滚已产生的内容数据 |

## 23. 外部依赖和待提供参数

实施不再存在阻塞性的产品决策。部署前由企业基础设施负责人提供：

- 可访问的 MySQL 8.0 实例和数据库凭证。
- 腾讯云短信应用、签名和模板参数。
- 私有 S3 兼容对象存储 Bucket、Endpoint、Region 和访问凭证；阿里云使用官方 S3 兼容 Endpoint。
- 生产域名、HTTPS 证书及反向代理或入口配置。

当前 V1 不接入企业身份源；后续若引入 OIDC、LDAP、CAS 或 SAML，需要新增认证适配和迁移设计，但内容、权限和 API 架构不受影响。
