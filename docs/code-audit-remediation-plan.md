# 代码审计分步修复计划

状态：已确认，实施中  
基线审计：[`code-audit-2026-07-21.md`](code-audit-2026-07-21.md)  
实施原则：一批一个主题、每批独立验收、每 3 至 5 批执行一次全量门禁

## 1. 目标与边界

本计划用于逐步修复全量代码审计发现的问题，避免同时修改授权、事务、Schema、前端状态和工程配置。每个批次必须能够独立合并和上线；后续批次未完成时，已完成批次仍应保持系统可用。

本轮不建设双人审批、异步导出任务或通用 Schema 数据迁移平台，不拆分整个 `content.Service`，也不把无关代码整理混入安全修复。

## 2. 权限模型决议

### 2.1 应急管理员

应急管理员是独立的 break-glass 身份，不是角色：

- 身份来自启用用户关联的有效本地凭证且 `emergency_admin=true`。
- 用于首次部署、短信服务不可用和权限配置损坏等特殊场景。
- 动态拥有全部当前及未来系统权限，以及全部 active 模型的所有模型权限。
- 权限不依赖角色，删除角色、解绑角色或清空角色授权都不能限制应急管理员。
- 只有应急管理员可以管理其他应急管理员的状态和角色。
- 系统始终至少保留一个启用的应急管理员。
- 应急凭证只能通过 `cms admin ensure` 和 `cms admin reset-password` 等运维命令创建或重置。

### 2.2 内置高危角色

系统内置唯一高危角色：

```text
key:          high_risk_admin
display_name: 高危管理员
kind:         high_risk
```

- 高危角色可分配给任意非应急用户，也可同时分配给多个用户。
- 高危角色动态拥有全部当前及未来系统权限，以及全部 active 模型权限。
- 高危角色定义不可编辑、不可删除，不能修改其系统或模型授权。
- 高危角色持有者可以给其他非应急用户授予或移除高危角色，也可以移除自己的高危角色。
- 高危角色持有者可以替换普通用户和其他高危角色用户的登录手机号。
- 高危角色不能禁用、改凭证、改角色或以其他方式管理任何应急管理员。
- 允许暂时没有用户持有高危角色，此时由应急管理员重新授予。
- 高危角色的授予、移除和高危凭证操作必须写入审计事件。

### 2.3 普通角色

- 普通用户权限严格等于全部普通角色授权的去重并集。
- 普通角色管理员只能委派自己拥有的系统权限和对应模型权限子集。
- 普通角色管理员不能修改自己的角色绑定或当前分配给自己的角色权限。
- 普通角色不能授予、移除或伪造高危角色身份。
- 即使某个普通角色静态包含全部权限码，也不能获得高危角色或应急管理员的身份能力。

### 2.4 高危操作矩阵

| 操作 | 普通角色管理员 | 高危角色 | 应急管理员 |
|---|---:|---:|---:|
| 管理普通用户 | 按静态权限 | 是 | 是 |
| 修改普通用户手机号 | 否 | 是 | 是 |
| 修改高危用户手机号 | 否 | 是 | 是 |
| 授予或移除高危角色 | 否 | 是 | 是 |
| 编辑或删除高危角色 | 否 | 否 | 否 |
| 管理应急管理员状态 | 否 | 否 | 是 |
| 修改应急管理员角色 | 否 | 否 | 是 |
| 修改应急管理员凭证 | 否 | 否 | 仅运维命令 |
| 授予普通权限 | 仅自身权限子集 | 不受子集限制 | 不受子集限制 |

## 3. 第一轮：权限基础与发布阻断项

### 批次 0：稳定前端测试门禁

- 修复测试结束后仍执行的 React scheduler、Promise、定时器或组件动画。
- 不通过增加固定等待时间掩盖问题。
- 清理 jsdom `getComputedStyle` 伪元素告警。
- `npm run test` 连续执行三次通过。

### 批次 1：建立主体安全等级

- `Principal` 增加 `is_emergency_admin` 和 `has_high_risk_role`。
- 权限计算顺序固定为：禁用用户无权限；应急管理员动态全权限；高危角色动态全权限；其他用户使用角色授权并集。
- 应急管理员和高危角色是两个互相独立的标记，应急身份优先。
- 同步 Go DTO、OpenAPI、前端类型和 mock。

### 批次 2：增加不可变内置高危角色

- 新增连续迁移，为 `roles` 增加 `kind`，保留并重命名旧库中可能存在的 `key=high_risk_admin` 自定义角色，再插入全新的唯一内置高危角色。历史角色继续保持普通权限和用户绑定，不因升级静默提权。
- 角色响应增加 `kind: custom | high_risk`。
- 服务端拒绝编辑、删除或替换高危角色授权，稳定错误码为 `builtin_role_immutable`。
- 迁移编号以实施时仓库中的下一连续编号为准，不预留或复用编号。

### 批次 3：实现高危角色授予边界

- 全量角色替换时比较原、新角色集合中的高危角色状态，分别鉴权授予和移除。
- 只有应急管理员或高危角色持有者可以授予或移除高危角色。
- 目标为应急管理员时，只有应急管理员可以修改其角色。
- 普通管理员不能通过省略高危角色 ID 绕过移除校验。
- 审计明确记录高危角色状态的 `from/to`。

第一轮完成后执行全量 Go、前端、双素材模式构建和 MySQL 8.0 迁移门禁，并进行一次授权专项代码审查。

## 4. 第二轮：权限安全约束

### 批次 4：普通角色委派子集限制

- 禁止普通管理员修改自己的角色绑定和当前分配给自己的角色权限。
- 普通管理员只能授予自身已有权限子集。
- 高危角色和应急管理员编辑普通角色时不受子集限制。
- 普通静态全权限角色仍不能获得高危身份能力。

### 批次 5：手机号管理规则

- 普通 `users.manage` 不能替换已激活登录手机号。
- 高危角色和应急管理员可以替换非应急用户手机号。
- 高危角色不能替换应急管理员手机号。
- 修改后撤销目标用户全部会话，审计只记录脱敏手机号。

### 批次 6：应急管理员账户保护

- 高危角色和普通角色都不能禁用、改凭证、改角色或改变应急管理员资格。
- 应急管理员可以管理其他应急管理员，但不能禁用最后一个有效应急管理员。
- 重置应急密码继续撤销该账户现有会话。

### 批次 7：历史 OIDC 会话

- 新增向前迁移撤销历史 OIDC 会话。
- 后续单独迁移将 Session 认证方式收窄到 `local/sms`。
- 运行时拒绝未知认证方式。

## 5. 第三轮：事务和数据一致性

### 批次 8：Schema 内容存在性查询使用当前事务

- `ContentExistenceChecker` 接收当前 `database.Querier`。
- 禁止事务内回退全局连接池。
- 增加小连接池并发测试。

### 批次 9：素材下载使用单一事务连接

- API Key 锁、scope 和素材引用验证使用同一事务执行器。
- 缩短 `FOR UPDATE` 持锁时间。
- 增加撤销与下载并发测试。

### 批次 10：阻止不兼容 Schema 收紧

- 有内容时拒绝可选改必填、长度收紧、数值范围收紧及等价不兼容变化。
- 允许显示信息修改和约束放宽。
- 使用稳定冲突错误提示先迁移数据。

### 批次 11：保护模型归档入站关联

- 归档前锁定并检查 active 入站关联字段。
- 存在引用时返回 `model_referenced`。
- 内容写入防御性要求关系目标模型 active。

### 批次 12：递归裁剪归档嵌套字段

- 按 active 字段树递归投影对象和重复组。
- 不修改历史 Revision，只调整客户端公开响应。

## 6. 第四轮：前端状态安全

### 批次 13：编辑器条目隔离与防重复保存

- 按 `modelId:entryId` 隔离草稿、字段有效性、事件游标和弹窗状态。
- 保存增加同步防重入锁和 `saving` 状态。
- 保存与工作流写操作互斥。

### 批次 14：未保存修改导航保护

- 接入 React Router blocker 和 `beforeunload`。
- 保存成功后解除保护。

### 批次 15：认证请求代次

- `authStore` 引入内部 epoch。
- 旧 Session 请求不能清空或覆盖新会话。
- 真实当前会话失效仍能正常退出。

## 7. 第五轮：协议与生产配置

### 批次 16：JSON 请求体上限

- Content 和 Schema Handler 使用 `http.MaxBytesReader`。
- 超限返回 413，并覆盖精确边界测试。

### 批次 17：工作流响应契约

- 工作流成功响应统一引用 `ContentEntry`。
- 增加实际响应对聚合 OpenAPI 的契约测试。

### 批次 18：收紧开发与数据库安全配置

- fixed SMS 只允许回环 Base URL 和监听地址。
- production 禁止 `MYSQL_ALLOW_INSECURE=true`。
- 同步 `.env.example` 和 README。

### 批次 19：可信代理客户端 IP

- 新增显式可信代理 CIDR 配置。
- 只有直接对端可信时解析标准代理头。
- 限流和审计统一使用同一客户端 IP 解析结果。

## 8. 第六轮：资源治理

### 批次 20：同步导出硬上限

CSV 保持同步实现，不恢复 Worker/Jobs：

- 每次最多 10,000 行。
- 临时 CSV 最大 50 MiB。
- 全进程最多 2 个并发导出。
- 同一用户最多 1 个并发导出。
- 单次生成和传输总期限 120 秒。
- 设置 `Content-Length`，检查并记录 `io.Copy` 错误。

这些值先作为后端常量；实际业务数据证明不合适时再单独配置化。

### 批次 21A：素材确认并发限制

- 限制全局和用户级素材确认并发。
- 对超大对象拒绝同步确认或降低生产硬上限。

### 批次 21B：内容 API Key 限流

- 增加 Key/IP 请求频率和并发保护。

### 批次 21C：认证状态 TTL 清理

- 分批清理过期 CAPTCHA、短信挑战、限流记录和会话。

## 11. 验证规则

每批至少运行受影响包的聚焦测试和静态检查。每完成 3 至 5 批执行：

```bash
go test -count=1 -timeout 300s ./...
go test -race -count=1 -timeout 360s ./...
go vet ./...
cd web && npm run lint
cd web && npm run typecheck
cd web && npm run test
cd web && npm run build
cd web && VITE_ASSETS_ENABLED=false npm run build
```

涉及迁移的批次必须用 MySQL 8.0 空库执行全部迁移，再执行第二次迁移验证幂等；涉及历史数据修正时还必须验证升级路径。

## 12. 风险与回滚

- 高危角色可以继续传播高危角色，这是明确接受的业务能力；任一高危账户被接管都可能扩大高危主体数量，因此所有授予、移除和凭证替换必须审计。
- 高危角色永远不能反向管理应急管理员，确保根恢复能力不受日常最高权限账户控制。
- 角色和 Principal 响应只做兼容性字段扩展；旧前端应能忽略新增字段。
- 新增迁移只向前演进，不修改已执行迁移。应用回滚不能依赖回滚数据库迁移。
- 每批独立合并；发生回归时只回退该批应用代码，已经执行的数据迁移按向前修复处理。
